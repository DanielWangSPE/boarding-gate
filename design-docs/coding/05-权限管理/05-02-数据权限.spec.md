# 05-02 数据权限 — 编码规范（Spec）

## 1. 数据模型

### 1.1 数据范围枚举

```java
public enum DataScope {
    ALL(1, "全部数据"),
    CUSTOM(2, "自定义"),
    DEPT(3, "本部门"),
    DEPT_TREE(4, "本部门及下级"),
    SELF(5, "仅本人");
}
```

优先级：ALL > CUSTOM / DEPT_TREE > DEPT > SELF（多角色取最大范围）。

### 1.2 关联表

数据权限配置存储在角色管理模块的表中（已在 04-01、04-02 spec 中定义）：

- `sys_role.data_scope`：角色的数据范围类型
- `sys_role_dept`：CUSTOM 范围时关联的部门列表

### 1.3 业务表约定

需要数据权限控制的业务表必须包含以下字段：

```sql
-- 用于行级数据权限过滤
dept_id     BIGINT       COMMENT '所属部门 ID（DEPT / DEPT_TREE / CUSTOM 场景使用）',
created_by  BIGINT       COMMENT '创建人 ID（SELF 场景使用）',
```

## 2. 行级数据权限实现

### 2.1 注解定义

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataPermission {
    /**
     * 业务表的部门字段别名，默认 "d.id"
     * 用于生成 dept_id IN (...) 条件
     */
    String deptAlias() default "d.id";

    /**
     * 业务表的创建人字段别名，默认 "u.created_by"
     * 用于生成 created_by = ? 条件
     */
    String userAlias() default "u.created_by";

    /**
     * 是否启用数据权限，默认 true
     * 设为 false 可临时关闭
     */
    boolean enabled() default true;
}
```

### 2.2 AOP 切面 / MyBatis 拦截器

```
方案一：MyBatis Interceptor（推荐）
  ├─ 拦截 Executor#query 方法
  ├─ 检查 Mapper 方法上是否有 @DataPermission 注解
  ├─ 有 → 计算用户数据范围，改写 SQL
  └─ 无 → 放行

方案二：AOP 切面 + ThreadLocal
  ├─ @DataPermission 标注在 Service 方法上
  ├─ 切面计算数据范围，写入 ThreadLocal
  ├─ MyBatis 插件从 ThreadLocal 读取条件拼接 SQL
  └─ finally 清除 ThreadLocal
```

**推荐方案一**，减少 ThreadLocal 管理，直接在 SQL 层处理。

### 2.3 SQL 改写逻辑

```
原始 SQL:
  SELECT u.* FROM sys_user u LEFT JOIN sys_user_dept ud ON u.id = ud.user_id
  WHERE u.status = 1

改写后（DEPT_TREE 场景）:
  SELECT u.* FROM sys_user u LEFT JOIN sys_user_dept ud ON u.id = ud.user_id
  WHERE u.status = 1
  AND ud.dept_id IN (
    SELECT id FROM sys_dept WHERE FIND_IN_SET(#{userDeptId}, ancestors) OR id = #{userDeptId}
  )

改写后（CUSTOM 场景）:
  SELECT u.* FROM sys_user u LEFT JOIN sys_user_dept ud ON u.id = ud.user_id
  WHERE u.status = 1
  AND ud.dept_id IN (10, 20, 30)

改写后（SELF 场景）:
  SELECT u.* FROM sys_user u LEFT JOIN sys_user_dept ud ON u.id = ud.user_id
  WHERE u.status = 1
  AND u.created_by = #{currentUserId}
```

### 2.4 多角色数据范围合并

```
计算用户最终数据范围
  │
  ├─ 1. 获取用户的所有正常角色
  ├─ 2. 提取所有角色的 data_scope
  ├─ 3. 按优先级合并
  │      ├─ 任一角色为 ALL → 最终 ALL（不追加过滤）
  │      ├─ 否则，收集 CUSTOM 的 deptIds + DEPT_TREE 展开的 deptIds + DEPT 的 deptId
  │      │      └─ 合并去重 → 最终 IN (...)
  │      └─ 仅 SELF → 最终 created_by = userId
  └─ 4. 缓存计算结果
```

**合并规则详解**：

| 角色A         | 角色B         | 最终结果                    |
| ------------- | ------------- | --------------------------- |
| ALL           | 任意          | ALL                         |
| DEPT          | DEPT_TREE     | DEPT_TREE（包含 DEPT）      |
| CUSTOM(10,20) | CUSTOM(20,30) | CUSTOM(10,20,30)            |
| DEPT          | CUSTOM(10,20) | 用户部门 ∪ {10,20}          |
| SELF          | DEPT          | DEPT（DEPT 包含 SELF 范围） |

### 2.5 缓存设计

| Key                       | 内容                      | TTL     |
| ------------------------- | ------------------------- | ------- |
| `datascope:user:{userId}` | 用户聚合后的数据范围 JSON | 30 分钟 |
| `dept:ancestors:{deptId}` | 部门祖先链                | 60 分钟 |
| `dept:children:{deptId}`  | 部门直接子部门 ID 列表    | 60 分钟 |

**缓存结构示例**：

```json
{
  "scope": "MERGED",
  "deptIds": [10, 20, 30, 31, 32],
  "includeSelf": true
}
```

**刷新时机**：

- 用户角色变更 → 清除 `datascope:user:{userId}`
- 角色数据权限变更 → 清除关联用户的 `datascope:user:{userId}`
- 部门结构变更 → 清除 `dept:ancestors:*`、`dept:children:*`

## 4. 列级数据权限（扩展）

### 4.1 设计思路

```
方案：注解 + 序列化拦截

@ColumnPermission("salary:view")  // 标注在字段上
private BigDecimal salary;

// 序列化时检查当前用户是否具有 salary:view 权限
// 无权限 → 字段不出现在 JSON 中，或返回 null / 脱敏值
```

### 4.2 实现方式

```
JSON 序列化拦截
  │
  ├─ 自定义 Jackson Serializer / Filter
  ├─ 序列化时检查字段上的 @ColumnPermission 注解
  ├─ 从当前用户权限集中检查是否包含该权限标识
  ├─ 有权限 → 正常序列化
  └─ 无权限 → 跳过字段 / 输出为 null / 输出脱敏值
```

### 4.3 前端配合

```
前端列级权限控制
  │
  ├─ 表格列定义中标记权限标识
  │      { prop: 'salary', label: '工资', perms: 'salary:view' }
  ├─ 表格渲染时通过 hasPermission 过滤列
  └─ 后端不返回的字段 + 前端不渲染的列 = 双重保障
```

> 列级权限为扩展功能，一期可仅通过 `@Desensitize` 注解做脱敏处理，二期再实现完整的列级权限控制。

## 5. 编码规范

### 5.1 Mapper 方法标注

```java
@DataPermission(deptAlias = "ud.dept_id", userAlias = "u.id")
List<SysUser> selectUserList(@Param("query") UserQuery query);

@DataPermission(deptAlias = "p.dept_id", userAlias = "p.created_by")
List<Project> selectProjectList(@Param("query") ProjectQuery query);

@DataPermission(enabled = false) // 特定查询关闭数据权限
SysUser selectById(@Param("id") Long id);
```

### 5.2 业务表设计规范

所有需要数据权限控制的业务表须包含：

- `dept_id BIGINT`：数据所属部门
- `created_by BIGINT`：数据创建人
- 在 `dept_id` 字段上建立索引

### 5.3 避免的反模式

| 反模式 | 正确做法 |
| --- | --- |
| 在业务代码中手动拼接 dept_id 条件 | 使用 @DataPermission 注解 + 拦截器自动注入 |
| SQL 中硬编码部门 ID 列表 | 通过数据权限计算动态注入 |
| 前端不做列隐藏（仅依赖后端） | 前后端双重控制 |
| 对所有查询都加数据权限 | 仅对列表查询加，详情/ID 查询通过业务逻辑校验 |

### 5.4 性能优化

- `FIND_IN_SET` 在 `ancestors` 字段上的查询，适用于部门层级不深的场景（≤ 10 级）
- 部门层级较深或数据量大时，使用 Redis 预计算部门 ID 列表，改为 `IN (...)` 查询
- `dept_id` 字段必须建索引
- 数据范围缓存减少重复计算
- 子查询改为 JOIN 提升性能（视实际 SQL 复杂度决定）

## 6. 错误码

| 错误码 | 说明                   |
| ------ | ---------------------- |
| A0550  | 数据权限范围无效       |
| A0551  | 自定义范围未指定部门   |
| A0552  | 指定部门不存在或已停用 |
| 403    | 无权访问该数据         |

## 7. 安全要点

| 项目         | 要求                                                   |
| ------------ | ------------------------------------------------------ |
| 强制生效     | 数据权限通过拦截器全局生效，不依赖业务代码             |
| 超管豁免     | 超级管理员 data_scope 固定为 ALL                       |
| SQL 注入防护 | deptAlias / userAlias 通过白名单校验，不允许任意字符串 |
| 缓存一致性   | 角色权限变更 / 部门结构变更后必须刷新相关缓存          |
| 审计追踪     | 记录数据权限配置的变更（通过角色管理模块审计）         |
| 列级权限     | 后端拦截序列化，前端隐藏列，双重保障                   |
