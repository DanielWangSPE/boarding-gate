# 05-01 功能权限 — 编码规范（Spec）

## 1. 数据模型

### 1.1 菜单/权限资源表（sys_menu）

```sql
CREATE TABLE sys_menu (
  id            BIGINT       NOT NULL AUTO_INCREMENT COMMENT '主键',
  parent_id     BIGINT       NOT NULL DEFAULT 0      COMMENT '父节点 ID，0 表示顶级',
  menu_name     VARCHAR(64)  NOT NULL                 COMMENT '菜单/按钮名称',
  menu_type     CHAR(1)      NOT NULL                 COMMENT '类型：M=目录, C=菜单, F=按钮',
  app_id        VARCHAR(64)  DEFAULT NULL             COMMENT '所属应用 ID，NULL 表示门户自身',
  path          VARCHAR(256) DEFAULT NULL             COMMENT '路由路径',
  component     VARCHAR(256) DEFAULT NULL             COMMENT '前端组件路径',
  perms         VARCHAR(128) DEFAULT NULL             COMMENT '权限标识，如 user:create',
  icon          VARCHAR(64)  DEFAULT NULL             COMMENT '菜单图标',
  sort_order    INT          NOT NULL DEFAULT 0       COMMENT '排序号',
  visible       TINYINT(1)   NOT NULL DEFAULT 1       COMMENT '是否可见：1=是, 0=否',
  status        TINYINT(1)   NOT NULL DEFAULT 1       COMMENT '状态：1=正常, 0=停用',
  remark        VARCHAR(256) DEFAULT NULL             COMMENT '备注',
  created_by    BIGINT       DEFAULT NULL             COMMENT '创建人',
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by    BIGINT       DEFAULT NULL             COMMENT '更新人',
  updated_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  INDEX idx_parent_id (parent_id),
  INDEX idx_app_id (app_id),
  UNIQUE INDEX uk_perms (perms)
) COMMENT '菜单与权限资源表';
```

**字段说明**：

| 字段 | 约束 | 说明 |
| --- | --- | --- |
| menu_type | M/C/F | M=目录（可包含子菜单）；C=菜单（对应一个路由页面）；F=按钮（对应一个操作权限） |
| perms | 全局唯一 | 格式 `模块:操作`，如 `user:create`，按钮类型必填，菜单/目录可不填 |
| app_id | 可为 NULL | NULL 代表门户自身的菜单，非 NULL 对应子应用 ID |
| visible | 布尔 | 控制菜单是否在侧边栏展示，不影响权限判断（如隐藏的内页路由） |

### 1.2 与角色关联

权限资源通过 `sys_role_menu` 表与角色关联（已在 04-02 角色权限配置 spec 中定义），此处不重复定义。

### 2.2 创建权限资源

```
POST /permissions
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "parentId": 100,
  "menuName": "删除用户",
  "menuType": "F",
  "appId": null,
  "perms": "user:delete",
  "sortOrder": 3
}
```

**处理流程**

```
创建权限资源
  │
  ├─ 1. 参数校验
  │      ├─ menuName 非空，长度 ≤ 64
  │      ├─ menuType 为 M / C / F 之一
  │      ├─ F 类型 perms 必填
  │      ├─ M / C 类型 path 必填
  │      └─ appId 非空时校验应用存在
  ├─ 2. parentId 校验
  │      ├─ parentId = 0 或指向已有节点
  │      └─ 父节点 menuType 不能为 F（按钮下不能挂节点）
  ├─ 3. perms 全局唯一校验
  ├─ 4. 同级下 menuName 唯一校验
  ├─ 5. INSERT sys_menu
  ├─ 6. 清除权限资源树缓存（permission:tree、permission:tree:{appId}）
  ├─ 7. 记录操作留痕
  └─ 8. 返回新增资源 ID
```

### 2.3 更新权限资源

```
PUT /permissions/{id}
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "menuName": "删除用户",
  "perms": "user:remove",
  "sortOrder": 4,
  "visible": true,
  "status": 1
}
```

**处理流程**

```
更新权限资源
  │
  ├─ 1. 校验资源存在
  ├─ 2. menuType 不可修改（防止类型变更导致结构混乱）
  ├─ 3. 如果修改了 perms，进行全局唯一校验
  ├─ 4. UPDATE sys_menu
  ├─ 5. 如果修改了 perms 或 status
  │      ├─ 查询关联此 menu 的所有角色 → 关联的所有用户
  │      └─ 批量清除用户权限缓存
  ├─ 6. 清除权限资源树缓存
  ├─ 7. 记录操作留痕
  └─ 8. 返回成功
```

### 2.4 删除权限资源

```
DELETE /permissions/{id}
Authorization: Bearer {accessToken}
```

**处理流程**

```
删除权限资源
  │
  ├─ 1. 校验资源存在
  ├─ 2. 如果存在子节点，拒绝删除（需先删子节点）
  ├─ 3. 删除 sys_role_menu 中引用此 ID 的关联
  ├─ 4. DELETE sys_menu
  ├─ 5. 查询受影响的用户，批量清除权限缓存
  ├─ 6. 清除权限资源树缓存
  ├─ 7. 记录操作留痕
  └─ 8. 返回成功
```

### 2.5 批量注册权限资源

```
POST /permissions/batch-register
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "appId": "project-management",
  "resources": [
    {
      "menuName": "项目管理",
      "menuType": "M",
      "path": "/project",
      "icon": "project",
      "sortOrder": 1,
      "children": [
        {
          "menuName": "项目列表",
          "menuType": "C",
          "path": "list",
          "component": "project/list/index",
          "perms": "project:list",
          "children": [
            {
              "menuName": "新建项目",
              "menuType": "F",
              "perms": "project:create"
            },
            {
              "menuName": "编辑项目",
              "menuType": "F",
              "perms": "project:update"
            },
            {
              "menuName": "删除项目",
              "menuType": "F",
              "perms": "project:delete"
            }
          ]
        }
      ]
    }
  ]
}
```

**处理流程**

```
批量注册
  │
  ├─ 1. 校验 appId 对应的应用存在
  ├─ 2. 递归校验 resources 树结构合法性
  │      ├─ perms 全局唯一（与已有 + 本批次内不重复）
  │      └─ menuType 层级合法（F 不能有 children）
  ├─ 3. 策略选择
  │      ├─ 增量模式（默认）：仅新增不存在的资源，已有资源跳过
  │      └─ 全量模式：先删除该 appId 下所有资源，再全量插入
  ├─ 4. 递归处理树结构，分配 parentId，批量 INSERT
  ├─ 5. 清除权限资源树缓存
  ├─ 6. 记录操作留痕
  └─ 7. 返回 { created: N, skipped: M }
```

### 2.6 获取指定应用的权限资源

```
GET /permissions/apps/{appId}
Authorization: Bearer {accessToken}
```

**Response**：结构同 2.1，仅包含指定应用的资源。

## 3. 权限标识体系

### 3.1 命名规范

```
格式: {module}:{action}

示例:
  user:list          用户列表查看
  user:create        创建用户
  user:update        编辑用户
  user:delete        删除用户
  user:export        导出用户
  user:import        导入用户
  role:list          角色列表
  role:assign        分配角色
  project:create     创建项目（子应用）
```

**规则**：

- 仅使用小写字母、数字和冒号
- `module` 为功能模块名，`action` 为操作名
- 冒号分隔，最多两级
- 子应用权限标识建议以子应用名为 module 前缀，如 `project:create`
- 权限标识全局唯一，不可跨应用重复

### 3.2 前后端一致性

```
前端                                     后端
──────────────────                       ──────────────────
v-permission="user:create"       ←→      @PreAuthorize("user:create")
<Authorized perms="user:delete"> ←→      @PreAuthorize("user:delete")
usePermission('user:export')     ←→      @PreAuthorize("user:export")
```

后端鉴权注解直接使用 `perms` 字段值，无需额外映射层。

## 4. 权限资源树构建

### 4.1 查询逻辑

```java
// 伪代码
List<SysMenu> allMenus = menuMapper.selectAll(appId, status);
List<MenuVO> tree = buildTree(allMenus, 0); // parentId = 0 为根

private List<MenuVO> buildTree(List<SysMenu> menus, Long parentId) {
    return menus.stream()
        .filter(m -> m.getParentId().equals(parentId))
        .sorted(Comparator.comparingInt(SysMenu::getSortOrder))
        .map(m -> {
            MenuVO vo = convert(m);
            vo.setChildren(buildTree(menus, m.getId()));
            return vo;
        })
        .collect(Collectors.toList());
}
```

### 4.2 缓存策略

| Key                       | 内容                  | TTL     |
| ------------------------- | --------------------- | ------- |
| `permission:tree`         | 完整的权限资源树 JSON | 60 分钟 |
| `permission:tree:{appId}` | 指定应用的权限资源树  | 60 分钟 |

**刷新时机**：权限资源增删改时清除。

## 5. 后端鉴权实现

### 5.1 鉴权流程

```
API 请求到达
  │
  ├─ 1. Gateway / Filter 校验 Token 合法性
  ├─ 2. 检查是否为白名单接口（如 GET/POST /auth/crypto/login）
  │      └─ 是 → 放行
  ├─ 3. 解析 Token 获取 userId
  ├─ 4. 从 Redis 缓存读取用户权限集
  │      └─ 缓存未命中 → 从 DB 计算并写入缓存
  ├─ 5. 检查方法上的 @PreAuthorize 注解
  │      ├─ 用户权限集包含 *:*:* → 放行（超管）
  │      ├─ 用户权限集包含所需标识 → 放行
  │      └─ 不包含 → 返回 403 Forbidden
  └─ 6. 执行业务逻辑
```

### 5.2 白名单配置

```yaml
security:
  whitelist:
    - /auth/crypto/login
    - /auth/refresh
    - /actuator/health
```

白名单接口无需携带 Token，以配置文件形式管理，支持通配符。

### 5.3 403 响应格式

```json
{
  "code": 403,
  "message": "无权访问该资源"
}
```

## 6. 前端权限控制

### 6.1 权限指令 v-permission

```vue
<!-- 单权限 -->
<el-button v-permission="'user:create'">新增用户</el-button>

<!-- 多权限（OR，满足其一即显示） -->
<el-button v-permission="['user:update', 'user:delete']">操作</el-button>
```

**实现要点**：

- 注册全局自定义指令
- 从权限 Store 获取当前用户权限集
- 判断是否命中 → 不命中时 `el.parentNode.removeChild(el)`
- 超管（权限集包含 `*:*:*`）直接通过

### 6.2 权限组件 Authorized

```vue
<Authorized perms="user:create">
  <el-button>新增用户</el-button>
</Authorized>
```

**实现要点**：

- 函数式组件，接收 `perms` 属性
- 判断逻辑同指令
- 不满足时渲染空节点（不是 `display:none`）

### 6.3 权限 Hook usePermission

```typescript
const { hasPermission } = usePermission();

if (hasPermission('user:export')) {
  // 显示导出按钮 / 执行导出逻辑
}
```

### 6.4 前端权限数据加载

```
用户登录成功 / 页面刷新
  │
  ├─ 1. 调用 GET /permissions/current → 用户权限标识集
  ├─ 2. 调用 GET /menus/user → 用户菜单树
  ├─ 3. 权限标识集存入 Pinia / Vuex Store（内存）
  ├─ 4. 菜单树动态注册路由
  └─ 5. 退出登录时清空 Store
```

## 7. 错误码

| 错误码 | 说明                         |
| ------ | ---------------------------- |
| A0500  | 权限资源不存在               |
| A0501  | 权限标识已存在（perms 重复） |
| A0502  | 存在子节点，不可删除         |
| A0503  | 父节点不存在或类型不合法     |
| A0504  | 按钮类型权限标识不能为空     |
| A0505  | 同级菜单名称重复             |
| A0506  | 应用不存在                   |
| 403    | 无权访问该资源               |

## 8. 安全要点

| 项目             | 要求                                            |
| ---------------- | ----------------------------------------------- |
| 权限管理         | 权限资源增删改需 `admin:permission:manage` 权限 |
| 白名单最小化     | 仅将必要的公共接口加入白名单                    |
| 权限标识不可伪造 | 前端权限仅做 UI 控制，后端必须独立鉴权          |
| 子应用隔离       | 批量注册时校验 appId 归属，防止越权             |
| 缓存一致性       | 权限资源变更后必须清除相关缓存                  |
| 审计追踪         | 记录所有权限资源的创建、修改、删除操作          |
