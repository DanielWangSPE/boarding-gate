# 05-03 权限查询与检验 — 编码规范（Spec）

## 1. 后端权限查询服务

### 1.1 接口规范

#### 1.1.1 获取当前用户完整权限集

```
GET /permissions/current
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "permissions": [
      "user:list",
      "user:create",
      "user:update",
      "user:delete",
      "role:list",
      "project:list",
      "project:create"
    ]
  }
}
```

**说明**：超级管理员返回 `["*:*:*"]`，前端据此判断为全权限。

#### 1.1.2 校验当前用户是否具有指定权限

```
POST /permissions/check
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "permissions": ["user:create", "user:delete"],
  "mode": "ANY"
}
```

| 参数        | 必填 | 说明                                   |
| ----------- | ---- | -------------------------------------- |
| permissions | 是   | 待校验的权限标识列表                   |
| mode        | 否   | ANY=满足其一即可（默认），ALL=全部满足 |

**Response 200**

```json
{
  "code": 200,
  "data": {
    "granted": true,
    "details": {
      "user:create": true,
      "user:delete": false
    }
  }
}
```

#### 1.1.3 获取当前用户数据权限范围

```
GET /permissions/data-scope
Authorization: Bearer {accessToken}
```

**Response**：参见 05-02 数据权限 spec 3.1 节。

### 1.2 权限计算流程

```
查询用户权限集
  │
  ├─ 1. 从 Token 解析 userId
  ├─ 2. 查询 Redis 缓存 permission:user:{userId}
  │      ├─ 命中 → 直接返回
  │      └─ 未命中 → 从 DB 计算
  ├─ 3. DB 计算流程
  │      ├─ 查询 sys_user_role → 获取用户的所有角色 ID（角色状态=正常）
  │      ├─ 查询 sys_role → 检查是否包含超管角色
  │      │      └─ 是 → 返回 ["*:*:*"]
  │      ├─ 查询 sys_role_menu → 获取所有角色关联的菜单 ID（合并去重）
  │      ├─ 查询 sys_menu → 获取菜单详情，提取 perms 字段（过滤 status=1）
  │      └─ 合并去重 → 用户权限标识集
  ├─ 4. 写入 Redis 缓存
  └─ 5. 返回
```

### 1.3 权限校验逻辑

```java
// 伪代码 — 权限校验核心方法
public boolean hasPermission(Long userId, String permission) {
    Set<String> userPerms = getUserPermissions(userId);
    if (userPerms.contains("*:*:*")) return true;
    return userPerms.contains(permission);
}

public boolean hasAnyPermission(Long userId, List<String> permissions) {
    Set<String> userPerms = getUserPermissions(userId);
    if (userPerms.contains("*:*:*")) return true;
    return permissions.stream().anyMatch(userPerms::contains);
}

public boolean hasAllPermissions(Long userId, List<String> permissions) {
    Set<String> userPerms = getUserPermissions(userId);
    if (userPerms.contains("*:*:*")) return true;
    return userPerms.containsAll(permissions);
}
```

## 2. 缓存设计

### 2.1 缓存 Key 汇总

| Key                        | 类型         | 内容                 | TTL     |
| -------------------------- | ------------ | -------------------- | ------- |
| `permission:user:{userId}` | Set          | 用户权限标识集合     | 30 分钟 |
| `menu:user:{userId}`       | String(JSON) | 用户可访问的菜单树   | 30 分钟 |
| `datascope:user:{userId}`  | String(JSON) | 用户聚合后的数据范围 | 30 分钟 |
| `permission:tree`          | String(JSON) | 完整权限资源树       | 60 分钟 |
| `permission:tree:{appId}`  | String(JSON) | 指定应用权限资源树   | 60 分钟 |

### 2.2 缓存刷新策略

| 触发事件 | 清除的缓存 Key |
| --- | --- |
| 角色功能权限变更 | 该角色关联的所有用户：`permission:user:{userId}`、`menu:user:{userId}` |
| 角色数据权限变更 | 该角色关联的所有用户：`datascope:user:{userId}` |
| 角色状态变更（停用/启用） | 该角色关联的所有用户：全部权限缓存 |
| 用户角色变更 | 该用户：全部权限缓存 |
| 权限资源增删改 | `permission:tree`、`permission:tree:{appId}` |
| 权限资源 perms / status 变更 | 受影响的用户：`permission:user:{userId}` |
| 部门结构变更 | `dept:ancestors:*`、`dept:children:*`，受影响用户：`datascope:user:{userId}` |

### 2.3 批量清除实现

```java
// 伪代码 — 使用 Redis Pipeline 批量清除
public void clearUserPermissionCache(List<Long> userIds) {
    if (userIds.isEmpty()) return;

    List<String> keys = new ArrayList<>();
    for (Long userId : userIds) {
        keys.add("permission:user:" + userId);
        keys.add("menu:user:" + userId);
        keys.add("datascope:user:" + userId);
    }

    // 分批处理，每批 500 个 key
    Lists.partition(keys, 500).forEach(batch -> {
        redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
            for (String key : batch) {
                connection.del(key.getBytes());
            }
            return null;
        });
    });
}
```

### 2.4 缓存穿透防护

- 用户无任何权限时，缓存空集合（`Set.of()`），不缓存 NULL
- 设置短 TTL（如 5 分钟）防止空值长期占用
- 不存在的用户 ID 不缓存，由上游 Token 校验保障

### 3.2 权限 Store

```typescript
// permission.ts
interface PermissionState {
  permissions: string[];
  menus: MenuTree[];
  loaded: boolean;
}

const usePermissionStore = defineStore('permission', {
  state: (): PermissionState => ({
    permissions: [],
    menus: [],
    loaded: false,
  }),

  getters: {
    hasPermission: (state) => (perm: string) => {
      if (state.permissions.includes('*:*:*')) return true;
      return state.permissions.includes(perm);
    },

    hasAnyPermission: (state) => (perms: string[]) => {
      if (state.permissions.includes('*:*:*')) return true;
      return perms.some((p) => state.permissions.includes(p));
    },
  },

  actions: {
    async loadPermissions() {
      const [permRes, menuRes] = await Promise.all([
        api.get('/permissions/current'),
        api.get('/menus/user'),
      ]);
      this.permissions = permRes.data.permissions;
      this.menus = menuRes.data;
      this.loaded = true;
    },

    clear() {
      this.permissions = [];
      this.menus = [];
      this.loaded = false;
    },
  },
});
```

### 3.3 权限指令

```typescript
// directive.ts
const vPermission: Directive = {
  mounted(el: HTMLElement, binding) {
    const store = usePermissionStore();
    const value = binding.value;

    const perms = Array.isArray(value) ? value : [value];
    const hasPermission = perms.some((p) => store.hasPermission(p));

    if (!hasPermission) {
      el.parentNode?.removeChild(el);
    }
  },
};
```

### 3.4 权限组件

```vue
<!-- Authorized.vue -->
<script setup lang="ts">
const props = defineProps<{
  perms: string | string[];
  mode?: 'any' | 'all';
}>();

const store = usePermissionStore();
const perms = computed(() =>
  Array.isArray(props.perms) ? props.perms : [props.perms],
);

const granted = computed(() => {
  if (props.mode === 'all') {
    return perms.value.every((p) => store.hasPermission(p));
  }
  return perms.value.some((p) => store.hasPermission(p));
});
</script>

<template>
  <slot v-if="granted" />
</template>
```

### 3.5 权限 Hook

```typescript
// usePermission.ts
export function usePermission() {
  const store = usePermissionStore();

  return {
    hasPermission: (perm: string) => store.hasPermission(perm),
    hasAnyPermission: (perms: string[]) => store.hasAnyPermission(perms),
    hasAllPermissions: (perms: string[]) =>
      perms.every((p) => store.hasPermission(p)),
    permissions: computed(() => store.permissions),
    isSuperAdmin: computed(() => store.permissions.includes('*:*:*')),
  };
}
```

## 4. 权限实时刷新

### 4.1 WebSocket 推送（可选）

```
权限变更实时通知
  │
  ├─ 后端：角色权限变更 → 查询受影响用户 → 通过 WebSocket 推送
  │      消息格式: { type: "PERMISSION_CHANGED", userId: 123 }
  ├─ 前端：收到消息 → 重新加载权限集
  │      permissionStore.loadPermissions()
  └─ 路由守卫：检测权限变更标记 → 刷新权限 → 重新过滤路由
```

### 4.2 轮询兜底

```
如果不使用 WebSocket：
  ├─ 前端定时请求 GET /permissions/current（间隔可配，默认 5 分钟）
  ├─ 对比本地权限集与返回的权限集
  │      ├─ 一致 → 无操作
  │      └─ 不一致 → 更新 Store，刷新路由
  └─ 仅在浏览器标签可见时轮询（visibilitychange 优化）
```

## 6. 编码规范

### 6.1 权限校验原则

| 原则             | 说明                                                    |
| ---------------- | ------------------------------------------------------- |
| 后端必须独立鉴权 | 不依赖前端传递权限信息，后端从 Token + 缓存/DB 独立判断 |
| 前端仅做 UI 控制 | 前端权限控制是体验优化，不是安全保障                    |
| 权限数据不可信   | 前端收到的权限列表不传回后端做鉴权依据                  |
| 失败安全         | 缓存异常时从 DB 重新计算，不默认放行                    |

### 6.2 SDK 设计原则

| 原则              | 说明                                              |
| ----------------- | ------------------------------------------------- |
| 权限数据单一来源  | 主应用加载，子应用不重复请求                      |
| 响应式更新        | 权限变更后 UI 自动更新（基于 Pinia 的响应式特性） |
| 零配置使用        | 子应用引入 SDK 后开箱即用                         |
| Tree-shaking 友好 | 支持按需导入，不增加无用包体积                    |

### 6.3 性能优化

| 优化项         | 方式                             |
| -------------- | -------------------------------- |
| 减少接口请求   | 登录后一次性加载，变更时增量刷新 |
| 缓存权限集     | Redis 缓存 + 前端内存缓存双层    |
| 批量清缓存     | Redis Pipeline 分批处理          |
| 权限集数据结构 | 使用 Set 而非 List，O(1) 查找    |
| 菜单树缓存     | 避免每次请求都重建树结构         |

## 7. 错误码

| 错误码 | 说明                                  |
| ------ | ------------------------------------- |
| 401    | 未登录或 Token 无效                   |
| 403    | 已登录但无权访问                      |
| A0560  | 权限查询参数无效                      |
| A0561  | 权限校验模式无效（mode 不是 ANY/ALL） |

## 8. 安全要点

| 项目           | 要求                                                 |
| -------------- | ---------------------------------------------------- |
| Token 校验前置 | 权限校验前必须先完成 Token 合法性校验                |
| 缓存安全       | Redis 权限缓存数据属于敏感信息，需限制访问           |
| SDK 通信安全   | 主应用到子应用的权限传递通过 props，不经过 URL       |
| 失败兜底       | 缓存/数据库异常时拒绝访问，不默认放行                |
| 权限集不落盘   | 前端权限集仅存内存（Pinia Store），不写 localStorage |
| 退出即清       | 用户退出登录时，前端清空所有权限数据                 |
