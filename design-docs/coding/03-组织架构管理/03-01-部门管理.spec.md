# 03-01 部门管理 — 编码规范（Spec）

## 1. 数据模型

### 1.1 部门表（sys_dept）

```sql
CREATE TABLE sys_dept (
  id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
  parent_id     BIGINT       NOT NULL DEFAULT 0 COMMENT '父部门 ID，根部门为 0',
  ancestors     VARCHAR(500) NOT NULL DEFAULT '' COMMENT '祖先路径，如 0,1,10',
  dept_name     VARCHAR(50)  NOT NULL COMMENT '部门名称',
  dept_code     VARCHAR(50)  NOT NULL COMMENT '部门编码',
  leader_id     BIGINT       NULL     COMMENT '负责人用户 ID',
  sort_order    INT          NOT NULL DEFAULT 0 COMMENT '排序号',
  status        TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '状态：0-停用 1-正常',
  remark        VARCHAR(500) NULL     COMMENT '备注',
  created_by    BIGINT       NULL,
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by    BIGINT       NULL,
  updated_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_dept_code (dept_code),
  INDEX idx_parent_id (parent_id),
  INDEX idx_ancestors (ancestors)
) COMMENT '部门表';
```

### 1.2 ancestors 字段说明

`ancestors` 存储从根到当前节点父级的完整路径，用逗号分隔：

```
根部门（id=1）       → ancestors = "0"
├─ 技术中心（id=10）  → ancestors = "0,1"
│  ├─ 研发部（id=100）→ ancestors = "0,1,10"
│  └─ 测试部（id=101）→ ancestors = "0,1,10"
└─ 业务中心（id=20）  → ancestors = "0,1"
```

用途：

- 查询某部门所有下级：`WHERE ancestors LIKE '0,1,10,%' OR id = 10`
- 查询某部门所有上级：解析 ancestors 字符串获取 ID 列表

## 2. 接口规范

### 2.1 获取部门树

```
GET /departments?status=1
Authorization: Bearer {accessToken}
```

**查询参数**

| 参数    | 必填 | 说明                                           |
| ------- | ---- | ---------------------------------------------- |
| status  | 否   | 筛选状态，不传则返回全部                       |
| keyword | 否   | 按部门名称模糊搜索（命中节点及其祖先路径展开） |

**Response 200**

```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "parentId": 0,
      "deptName": "XX公司",
      "deptCode": "ROOT",
      "leaderName": "张总",
      "sortOrder": 0,
      "status": 1,
      "children": [
        {
          "id": 10,
          "parentId": 1,
          "deptName": "技术中心",
          "deptCode": "TECH",
          "leaderName": "李总",
          "sortOrder": 1,
          "status": 1,
          "children": []
        }
      ]
    }
  ]
}
```

**实现要点**：

- 一次查询全量部门列表（通常不超过几百条），在内存中构建树形结构
- 按 `sort_order ASC` 排序
- 搜索时：匹配的节点 + 其祖先链路全部返回，前端展开命中路径

### 2.2 创建部门

```
POST /departments
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "parentId": 1,
  "deptName": "研发部",
  "deptCode": "RD",
  "leaderId": 100,
  "sortOrder": 1,
  "remark": ""
}
```

**Response 错误码**

| 业务 code | 说明                 |
| --------- | -------------------- |
| A0400     | 部门编码已存在       |
| A0401     | 父部门不存在或已停用 |
| A0402     | 同级部门名称重复     |

**处理流程**

```
创建部门
  │
  ├─ 1. 校验 deptCode 唯一性
  ├─ 2. 校验父部门存在且状态正常
  ├─ 3. 校验同级部门名称不重复
  ├─ 4. 计算 ancestors = 父部门 ancestors + "," + 父部门 id
  ├─ 5. 写入 sys_dept
  ├─ 6. 记录操作留痕（异步）
  └─ 7. 返回新部门 ID
```

### 2.3 更新部门

```
PUT /departments/{id}
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "deptName": "研发一部",
  "leaderId": 101,
  "sortOrder": 2,
  "remark": "更新备注"
}
```

**说明**：

- `deptCode` 创建后不可修改，接口不接受此字段
- `parentId` 不可通过此接口修改，调整层级使用移动接口

### 2.4 删除部门

```
DELETE /departments/{id}
Authorization: Bearer {accessToken}
```

**Response 错误码**

| 业务 code | 说明                         |
| --------- | ---------------------------- |
| A0410     | 根部门不可删除               |
| A0411     | 该部门下存在子部门，不可删除 |
| A0412     | 该部门下存在用户，不可删除   |

### 2.5 启用/停用部门

```
PUT /departments/{id}/status
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "status": 0
}
```

**停用处理流程**

```
停用部门
  │
  ├─ 1. 校验非根部门
  ├─ 2. 查询所有下级部门（ancestors LIKE）
  ├─ 3. 批量更新本部门及所有下级部门 status = 0
  ├─ 4. 记录操作留痕
  └─ 5. 返回成功（附被级联停用的部门数量）
```

**启用处理**：

- 仅启用当前部门，不级联启用下级
- 启用前校验父部门状态为正常（父部门停用时不可启用子部门）

### 2.6 移动部门

```
PUT /departments/{id}/move
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "targetParentId": 20,
  "sortOrder": 3
}
```

**Response 错误码**

| 业务 code | 说明                             |
| --------- | -------------------------------- |
| A0410     | 根部门不可移动                   |
| A0420     | 目标父部门不存在或已停用         |
| A0421     | 不可移动到自身或自身的下级部门下 |

**处理流程**

```
移动部门
  │
  ├─ 1. 校验非根部门
  ├─ 2. 校验目标父部门存在且正常
  ├─ 3. 防循环：目标父部门的 ancestors 中不包含当前部门 ID
  ├─ 4. 计算新 ancestors
  ├─ 5. 更新当前部门的 parent_id、ancestors、sort_order
  ├─ 6. 批量更新所有下级部门的 ancestors（替换前缀）
  │      UPDATE sys_dept
  │      SET ancestors = REPLACE(ancestors, '旧前缀', '新前缀')
  │      WHERE ancestors LIKE '旧前缀,%'
  ├─ 7. 记录操作留痕
  └─ 8. 返回成功
```

## 3. 编码规范

### 3.1 树形结构构建

```java
// 伪代码：从扁平列表构建树
public List<DeptTreeNode> buildTree(List<SysDept> deptList) {
    Map<Long, DeptTreeNode> nodeMap = new LinkedHashMap<>();
    deptList.forEach(dept -> nodeMap.put(dept.getId(), toNode(dept)));

    List<DeptTreeNode> roots = new ArrayList<>();
    nodeMap.values().forEach(node -> {
        if (node.getParentId() == 0) {
            roots.add(node);
        } else {
            DeptTreeNode parent = nodeMap.get(node.getParentId());
            if (parent != null) {
                parent.getChildren().add(node);
            }
        }
    });
    return roots;
}
```

- 全量查询后在内存中构建树，不使用递归 SQL
- 部门数据量通常较小（百级别），全量加载无性能问题
- 部门树可缓存（Redis，TTL 10 分钟），部门变更时主动清除

### 3.2 ancestors 维护

- 创建时：`新ancestors = 父部门ancestors + "," + 父部门id`
- 移动时：更新自身和所有下级的 ancestors（使用 `REPLACE` 批量替换前缀）
- 删除时：无需处理（因为有子部门时不允许删除）
- ancestors 字段必须在事务中与 parent_id 同步更新，保持一致性

### 3.3 排序规则

- 同级部门按 `sort_order ASC` 排序
- 拖拽排序时前端传递完整的同级排序列表，后端批量更新 `sort_order`
- 新增部门默认 `sort_order = 当前同级最大值 + 1`

### 3.4 缓存策略

| Key | 内容 | TTL | 刷新时机 |
| --- | --- | --- | --- |
| `dept:tree` | 完整部门树 JSON | 10 分钟 | 部门增删改移动时清除 |
| `dept:children:{deptId}` | 指定部门的所有下级 ID 列表 | 10 分钟 | 部门变更时清除 |

### 3.5 事务边界

- 创建/更新/删除：单条操作，简单事务
- 停用级联：本部门 + 所有下级在同一事务中更新
- 移动部门：本部门 + 所有下级 ancestors 更新在同一事务中

## 4. 安全要点

| 项目       | 要求                                                   |
| ---------- | ------------------------------------------------------ |
| 权限控制   | 部门增删改需 `admin:dept:manage` 权限                  |
| 根部门保护 | 硬编码保护根部门，不可删除/停用/移动                   |
| 循环检测   | 移动操作必须检测循环引用                               |
| 操作留痕   | 所有部门变更操作记录日志，移动需记录变更前后的完整状态 |
| 缓存一致性 | 部门变更后清除部门树缓存和下级 ID 缓存                 |
