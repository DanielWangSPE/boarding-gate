# 03-02 部门成员管理 — 编码规范（Spec）

## 1. 数据模型

复用用户关联关系模块的 `sys_user_dept` 表：

```sql
-- 参见 02-03-用户关联关系.spec.md
CREATE TABLE sys_user_dept (
  id              BIGINT    PRIMARY KEY AUTO_INCREMENT,
  user_id         BIGINT    NOT NULL,
  dept_id         BIGINT    NOT NULL,
  created_at      DATETIME  NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE INDEX uk_user_dept_user (user_id),
  INDEX idx_dept_id (dept_id)
);
```

> 本模块与 `02-03 用户关联关系` 操作同一张表，区别仅在于操作视角不同。

## 2. 接口规范

### 2.1 获取部门成员列表

```
GET /departments/{id}/members?keyword=&includeChildren=false&page=1&pageSize=20
Authorization: Bearer {accessToken}
```

**查询参数**

| 参数            | 必填 | 说明                             |
| --------------- | ---- | -------------------------------- |
| keyword         | 否   | 按用户名/姓名模糊搜索            |
| includeChildren | 否   | 是否包含下级部门成员，默认 false |
| page            | 否   | 页码，默认 1                     |
| pageSize        | 否   | 每页条数，默认 20                |

**Response 200**

```json
{
  "code": 200,
  "data": {
    "total": 35,
    "page": 1,
    "pageSize": 20,
    "list": [
      {
        "userId": 1,
        "username": "zhangsan",
        "nickname": "张三",
        "employeeNo": "EMP001",
        "status": 1,
        "isLeader": true,
        "joinedAt": "2026-01-15T10:00:00Z"
      }
    ]
  }
}
```

**SQL 查询逻辑**

```sql
-- includeChildren = false 时
SELECT u.id, u.username, u.nickname, u.employee_no, u.status,
       ud.created_at AS joined_at,
       (d.leader_id = u.id) AS is_leader
FROM sys_user_dept ud
JOIN sys_user u ON ud.user_id = u.id AND u.deleted = 0
JOIN sys_dept d ON ud.dept_id = d.id
WHERE ud.dept_id = ?
ORDER BY u.created_at ASC

-- includeChildren = true 时
WHERE ud.dept_id IN (
  SELECT id FROM sys_dept WHERE id = ? OR ancestors LIKE CONCAT(?, '%')
)
```

### 2.2 设置部门成员归属

```
POST /departments/{id}/members
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "userIds": [10, 11, 12]
}
```

| 字段    | 说明                           |
| ------- | ------------------------------ |
| userIds | 要设置到当前部门的用户 ID 列表 |

**Response 200**

```json
{
  "code": 200,
  "data": {
    "updatedCount": 3,
    "skippedCount": 0,
    "skipped": []
  }
}
```

**Response 错误码**

| 业务 code | 说明                                 |
| --------- | ------------------------------------ |
| A0430     | 部门不存在或已停用                   |
| A0431     | 用户不存在或已停用                   |
| A0432     | 用户是其他部门负责人，请先调整负责人 |

**处理流程**

```text
设置成员归属
  │
  ├─ 1. 校验部门存在且状态正常
  ├─ 2. 校验所有 userIds 存在且未注销
  ├─ 3. 过滤已归属到当前部门的用户（跳过，不报错）
  ├─ 4. 检查这些用户是否仍担任其他部门负责人
  ├─ 5. 批量更新或替换这些用户在 sys_user_dept 中的唯一归属记录
  ├─ 6. 记录操作留痕（异步）
  └─ 7. 返回结果
```

### 2.3 设置部门负责人

```
PUT /departments/{id}/leader
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "leaderId": 100
}
```

传 `null` 表示取消负责人。

**Response 错误码**

| 业务 code | 说明                 |
| --------- | -------------------- |
| A0450     | 该用户不是本部门成员 |

**实现要点**：

- 更新 `sys_dept.leader_id`
- 负责人必须是该部门的成员（存在于 `sys_user_dept`）
- 一个部门只能有一个负责人

## 3. 编码规范

### 3.1 与用户模块的关系

- 本模块操作 `sys_user_dept` 表，与用户关联关系模块操作同一张表
- **部门视角**：在部门详情页中管理选中部门下的成员
- **用户视角**：在用户详情页中管理该用户所属的唯一组织节点
- 两个模块都需遵守“一个用户仅归属一个组织节点”的规则

### 3.2 批量操作性能

- 设置成员归属时可按批处理，每批 100 个用户
- 更新前先查询当前归属，避免无效写入
- 批量更新完成后统一清理相关缓存

### 3.3 事件发布

- 设置成员归属时发布 `DeptMemberChangedEvent`
- 事件数据：`{ deptId, userIds, operator }`
- 监听方：权限模块可根据部门变更刷新数据权限缓存

## 4. 安全要点

| 项目       | 要求                                     |
| ---------- | ---------------------------------------- |
| 权限控制   | 成员管理需 `admin:dept:member` 权限      |
| 负责人设置 | 需 `admin:dept:manage` 权限              |
| 数据权限   | 查看成员列表受当前用户的数据权限范围约束 |
| 操作留痕   | 设置成员归属、设置负责人均需记录操作留痕 |
