# 07-03 应用间通信 — 编码规范（Spec）

## 1. 通信方式概览

micro-app 提供两种内置通信机制：

| 方式 | 适用场景 | 方向 |
| --- | --- | --- |
| 数据通信（setData/getData） | 持续性状态传递（Token、权限、主题） | 主应用 → 子应用 |
| 事件通信（dispatch/addDataListener） | 一次性事件通知（路由变更、操作请求） | 子应用 → 主应用 |

## 2. 通信协议定义

### 2.1 主应用下发数据结构

```typescript
// 主应用传递给子应用的数据（通过 setData）
interface PortalData {
  token: string;
  userInfo: {
    userId: number;
    username: string;
    nickname: string;
    avatar: string;
  };
  permissions: string[];
  theme: {
    mode: 'light' | 'dark';
    primaryColor: string;
  };
}
```

### 2.2 子应用上报事件结构

```typescript
// 子应用通过 dispatch 发送的事件
interface SubAppEvent {
  type: SubAppEventType;
  payload?: any;
}

type SubAppEventType =
  | 'route-change' // 子应用内部路由变更
  | 'title-change' // 修改页面标题
  | 'open-dialog' // 请求主应用打开全局弹窗
  | 'app-error' // 子应用运行时错误上报
  | 'logout'; // 子应用请求登出

// 各事件 payload 定义
interface RouteChangePayload {
  path: string; // 子应用内部路径
}

interface TitleChangePayload {
  title: string;
}

interface AppErrorPayload {
  message: string;
  stack?: string;
  info?: string;
}
```

### 2.3 主应用通知子应用事件结构

```typescript
// 主应用通过 setData 向子应用下发的控制指令
interface PortalCommand {
  type: PortalCommandType;
  payload?: any;
  timestamp: number;
}

type PortalCommandType =
  | 'logout' // 通知子应用清理并登出
  | 'token-updated' // Token 刷新
  | 'theme-change' // 主题切换
  | 'permissions-updated' // 权限变更
  | 'locale-change'; // 语言切换
```

## 3. 主应用 → 子应用

### 3.1 初始数据传递

```vue
<!-- 通过 :data 属性绑定，响应式更新 -->
<micro-app name="project" url="..." :data="appData" />

<script setup>
const appData =
  computed <
  PortalData >
  (() => ({
    token: userStore.token,
    userInfo: userStore.userInfo,
    permissions: permissionStore.permissions,
    theme: settingsStore.theme,
  }));
</script>
```

数据变更时 micro-app 自动通知子应用。

### 3.2 命令式数据更新

```typescript
import microApp from '@micro-zoe/micro-app';

// Token 刷新后通知子应用
function onTokenRefreshed(newToken: string) {
  const apps = menuStore.loadedApps;
  apps.forEach((appId) => {
    microApp.setData(appId, {
      token: newToken,
      __command: { type: 'token-updated', timestamp: Date.now() },
    });
  });
}

// 主题切换
function onThemeChange(theme: ThemeConfig) {
  const apps = menuStore.loadedApps;
  apps.forEach((appId) => {
    microApp.setData(appId, {
      theme,
      __command: { type: 'theme-change', timestamp: Date.now() },
    });
  });
}

// 登出通知
function onLogout() {
  const apps = menuStore.loadedApps;
  apps.forEach((appId) => {
    microApp.setData(appId, {
      __command: { type: 'logout', timestamp: Date.now() },
    });
  });
}
```

### 3.3 子应用接收数据

```typescript
// 子应用入口
function setupPortalDataListener() {
  // 获取初始数据
  const data = window.microApp?.getData() as PortalData;
  if (data) {
    applyPortalData(data);
  }

  // 监听数据变更
  window.microApp?.addDataListener(
    (data: PortalData & { __command?: PortalCommand }) => {
      applyPortalData(data);

      // 处理指令
      if (data.__command) {
        handleCommand(data.__command);
      }
    },
  );
}

function applyPortalData(data: PortalData) {
  // 更新本地 Token
  localAuthStore.setToken(data.token);
  // 更新用户信息
  localUserStore.setUserInfo(data.userInfo);
  // 更新权限
  localPermissionStore.setPermissions(data.permissions);
  // 更新主题
  localThemeStore.setTheme(data.theme);
}

function handleCommand(command: PortalCommand) {
  switch (command.type) {
    case 'logout':
      localAuthStore.clear();
      break;
    case 'token-updated':
      // Token 已在 applyPortalData 中更新
      break;
    case 'theme-change':
      // 主题已在 applyPortalData 中更新
      break;
  }
}
```

## 4. 子应用 → 主应用

### 4.1 子应用发送事件

```typescript
// 子应用通用通信工具
class PortalBridge {
  static dispatch(event: SubAppEvent) {
    if (window.__MICRO_APP_ENVIRONMENT__) {
      window.microApp.dispatch(event);
    }
  }

  static routeChange(path: string) {
    this.dispatch({ type: 'route-change', payload: { path } });
  }

  static titleChange(title: string) {
    this.dispatch({ type: 'title-change', payload: { title } });
  }

  static reportError(error: AppErrorPayload) {
    this.dispatch({ type: 'app-error', payload: error });
  }

  static requestLogout() {
    this.dispatch({ type: 'logout' });
  }
}
```

### 4.2 主应用监听子应用事件

```vue
<micro-app name="project" @datachange="handleSubAppEvent" />

<script setup>
function handleSubAppEvent(e: CustomEvent) {
  const event = e.detail.data as SubAppEvent

  switch (event.type) {
    case 'route-change':
      // 同步浏览器 URL
      const fullPath = `${app.routePrefix}${event.payload.path}`
      router.replace(fullPath)
      break

    case 'title-change':
      // 更新标签页标题
      tabStore.updateTabTitle(currentPath, event.payload.title)
      break

    case 'app-error':
      // 上报到统一留痕与埋点通道
      logService.reportError({
        source: 'sub-app',
        appId: app.appId,
        ...event.payload
      })
      break

    case 'logout':
      // 子应用请求登出 → 执行全局登出
      userStore.logout()
      break
  }
}
</script>
```

## 5. 子应用 → 子应用

### 5.1 通过主应用中转

```
子应用 A 需要通知子应用 B
  │
  ├─ 1. 子应用 A: window.microApp.dispatch({ type: 'cross-app', target: 'app-b', data: {...} })
  ├─ 2. 主应用监听到 cross-app 事件
  ├─ 3. 主应用调用 microApp.setData('app-b', { __crossAppMessage: {...} })
  └─ 4. 子应用 B 通过 addDataListener 接收消息
```

### 5.2 跨应用路由跳转

```typescript
// 子应用内跳转到另一个子应用的页面
function navigateToApp(targetAppId: string, path: string) {
  if (window.__MICRO_APP_ENVIRONMENT__) {
    window.microApp.dispatch({
      type: 'cross-app-navigate',
      payload: { targetAppId, path },
    });
  }
}

// 主应用处理跨应用路由跳转
function handleCrossAppNavigate(payload: {
  targetAppId: string;
  path: string;
}) {
  const app = menuStore.userApps.find((a) => a.appId === payload.targetAppId);
  if (app) {
    router.push(`${app.routePrefix}${payload.path}`);
  }
}
```

## 7. 编码规范

### 7.1 通信原则

| 原则       | 说明                                                     |
| ---------- | -------------------------------------------------------- |
| 类型安全   | 所有通信数据必须有 TypeScript 类型定义                   |
| 单向数据流 | 全局状态从主应用流向子应用，子应用通过事件请求变更       |
| 事件命名   | 使用 kebab-case，如 `route-change`、`theme-change`       |
| 幂等处理   | 子应用接收数据时需幂等处理，同一数据多次推送不产生副作用 |
| 版本兼容   | 通信协议变更需向后兼容，新增字段不影响旧版子应用         |

### 7.2 避免的反模式

| 反模式                         | 正确做法                                 |
| ------------------------------ | ---------------------------------------- |
| 子应用直接操作主应用 DOM       | 通过事件通信请求主应用操作               |
| 子应用直接修改 window 全局状态 | 通过沙箱隔离的 window 操作               |
| 子应用间直接通信               | 通过主应用中转                           |
| 在通信数据中传递函数/DOM 引用  | 仅传递可序列化的数据                     |
| 监听事件不清理                 | micro-app 自动清理，自定义监听需手动清理 |

## 8. 安全要点

| 项目       | 要求                                                        |
| ---------- | ----------------------------------------------------------- |
| Token 传递 | Token 通过 setData 在内存中传递，不经过 URL 或 localStorage |
| 数据校验   | 主应用接收子应用事件时需校验数据合法性                      |
| 跨应用隔离 | 子应用无法读取其他子应用的通信数据                          |
| 敏感信息   | 通信数据中不包含密码等敏感信息                              |
| 日志脱敏   | 通信日志记录时对 Token 等字段脱敏                           |
