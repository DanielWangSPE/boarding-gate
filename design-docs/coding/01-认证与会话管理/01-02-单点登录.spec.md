# 01-02 单点登录（SSO） — 编码规范（Spec）

## 1. 方案选型

采用基于 Token 的 SSO 方案，适配当前内网部署和微前端主应用 + 子应用的运行模式。

### 1.1 当前方案

当前方案下：

- Token 存储在主应用的内存（运行时状态）中
- 子应用通过微前端通信机制从主应用获取 Token
- 子应用发起 API 请求时在 Header 中携带 Token

## 2. Token 传递机制

### 2.1 主应用 → 子应用

```
主应用
  │
  ├─ 子应用 mount 时，通过 props 传递：
  │    props: {
  │      token: {
  │        accessToken: "xxx",
  │        expiresIn: 1800
  │      },
  │      userInfo: { userId, username, ... },
  │      onTokenRefreshed: (newToken) => void,
  │      onLogout: () => void
  │    }
  │
  └─ Token 刷新后，通过 update 钩子或全局状态通知子应用更新
```

### 2.2 子应用 → API 网关

```
子应用 HTTP 请求
  │
  └─ Request Header:
       Authorization: Bearer {accessToken}
```

### 2.3 Token 刷新联动

```
子应用 API 请求
  │
  ├─ 收到 401（Token 过期）
  │    └─ 调用主应用提供的 refreshToken 方法
  │         ├─ 刷新成功 → 主应用广播新 Token → 已加载子应用更新
  │         └─ 刷新失败 → 主应用触发统一登出
  │
  └─ 主应用主动刷新（定时器 / 拦截器）
       └─ 刷新成功 → 通过全局状态 / 事件广播通知已加载子应用
```

## 3. 统一登出流程

```
触发登出（主应用菜单点击 / 子应用调用 onLogout）
  │
  ├─ 1. 主应用调用 POST /auth/logout
  │      ├─ 后端将当前会话对应的 Token 加入黑名单
  │      └─ 后端清除当前会话记录
  │
  ├─ 2. 主应用广播登出事件给所有已加载的子应用
  │      └─ 各子应用清除内存中的 Token 和用户状态
  │
  ├─ 3. 主应用清除自身状态
  │      └─ 清除内存中的 Token、用户信息、权限、菜单缓存
  │
  └─ 4. 跳转到登录页
```

## 4. 接口规范

### 4.1 统一登出

```
POST /auth/logout
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "message": "登出成功"
}
```

### 4.2 获取当前用户信息

```
GET /auth/userinfo
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "userId": "string",
    "username": "string",
    "nickname": "string",
    "avatar": "string",
    "roles": ["string"],
    "permissions": ["string"]
  }
}
```

## 5. 编码规范

### 5.1 Token 存储安全

- **禁止**将 Token 存入 `localStorage` 或 `sessionStorage`（防 XSS 窃取）
- Token 存储在 JavaScript 运行时内存中（如 Pinia/Vuex store、React Context）
- 页面刷新时通过 Refresh Token（HttpOnly Cookie）重新获取 Access Token

### 5.2 Refresh Token 存储

- Refresh Token 通过登录接口的 `Set-Cookie` 响应头写入 HttpOnly + Secure + SameSite=Strict 的 Cookie
- 前端 JS 无法读取 Refresh Token，仅由浏览器自动携带至 `/auth/refresh` 接口

### 5.3 子应用接入规范

- 子应用 **不应** 自行管理登录态，必须依赖主应用下发的 Token
- 子应用 HTTP 拦截器中需统一处理：
  - 请求拦截：从 props/全局状态中获取最新 Token 注入 Header
  - 响应拦截：收到 401 时调用主应用的 Token 刷新方法，而非自行跳转登录页
- 子应用 **禁止** 直接跳转至登录页，登出逻辑统一委托给主应用

### 5.4 通信事件定义

| 事件名 | 方向 | 触发时机 | 数据 |
| --- | --- | --- | --- |
| `portal:token-updated` | 主 → 子 | Token 刷新成功 | `{ accessToken, expiresIn }` |
| `portal:logout` | 主 → 子 | 主应用发起登出 | — |
| `app:request-logout` | 子 → 主 | 子应用请求登出 | — |
| `app:token-expired` | 子 → 主 | 子应用检测到 Token 过期 | — |

### 5.5 防重复刷新

- 多个子应用可能同时检测到 Token 过期并请求刷新
- 主应用需实现 Token 刷新锁（Promise 复用），同一时刻仅发起一次刷新请求
- 其他刷新请求等待同一个 Promise 结果

```typescript
// 伪代码示例：Token 刷新锁
let refreshPromise: Promise<Token> | null = null;

async function refreshToken(): Promise<Token> {
  if (refreshPromise) return refreshPromise;

  refreshPromise = doRefresh().finally(() => {
    refreshPromise = null;
  });

  return refreshPromise;
}
```

## 6. 安全要点

| 项目 | 要求 |
| --- | --- |
| Access Token 存储 | 仅存内存，禁止持久化到 Storage |
| Refresh Token 存储 | HttpOnly Cookie，JS 不可读取 |
| Token 吊销 | 登出时服务端将 Token 加入 Redis 黑名单，TTL = 剩余有效期 |
| 主子应用通信 | 使用微前端 props 或约定事件传递 Token，不明文持久化到浏览器存储 |
| CSRF 防护 | Refresh Token Cookie 设置 SameSite=Strict |
