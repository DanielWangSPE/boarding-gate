# 01-01 登录认证 — 编码规范（Spec）

## 1. 数据模型与范围

本阶段登录认证 **不** 使用 Redis（或其它存储）记录**单用户登录失败次数**，**不** 对账号实施**登录失败锁定**。

图形验证码亦不纳入本阶段；若后续需要失败计数、锁定、人机验证等能力，再单独补充 Spec 与数据模型。

**可选实现（推荐）**：为绑定「一次一密」的 SM4 参数，可使用 Redis 等短期存储：

```
Key:   auth:crypto:login:{cryptoId}
Type:  String 或 Hash（序列化后的密钥与 IV 等，或仅存服务端可解析的引用）
TTL:   与接口返回的 ttlSeconds 一致（建议 120～300 秒）
```

登录成功或解密失败后应删除该 Key，使 `cryptoId` **一次性**失效；若未使用 Redis，也须在进程内以等价方式保证 `cryptoId` 单次使用与过期。

## 2. 接口规范

本节 JSON 响应均落在统一外壳 **`ApiResponse`**（与后端 `com.boardinggate.web.dto.ApiResponse` 一致）：

- **`code`**：**字符串**。成功为 **`"200"`**；失败为 **`"A0210"`** 等业务码（勿使用 JSON 数字 `200`，避免与 HTTP 状态码混淆）。
- **`message`**：人类可读说明；成功时常见为 `"ok"`。
- **`data`**：业务载荷；失败时可为 `null`。
- **`error`**：失败时的附加信息；成功时为 `null`。

前端判断成功须使用 **`code === '200'`**（或等价字符串比较），勿与数字 `200` 比较。

### 2.1 获取登录 SM4 加密参数

用于登录前拉取本次会话的 SM4 密钥与 IV（及模式约定），与 `POST /auth/crypto/login` 中的 `cryptoId` 一一对应。

```
GET /auth/crypto/login
```

**Response 200**

```json
{
  "code": "200",
  "message": "ok",
  "data": {
    "cryptoId": "uuid-string",
    "algorithm": "SM4",
    "mode": "CBC",
    "key": "Base64(16-byte-key)",
    "iv": "Base64(16-byte-iv-for-CBC)",
    "ttlSeconds": 120
  },
  "error": null
}
```

**字段说明**

| 字段 | 说明 |
| --- | --- |
| `cryptoId` | 本次加密上下文的唯一标识；`POST /auth/crypto/login` 必须原样回传，服务端据此取回密钥材料并校验未过期、未重复使用。 |
| `algorithm` | 固定为 `SM4`（便于前端与网关扩展其它算法时保持兼容字段）。 |
| `mode` | 分组模式，本阶段约定 **`CBC`** 或 **`GCM`** 二选一（全栈实现须一致）；若 `GCM`，`iv` 位置填 **Nonce**（长度按所选库与国密实现约定，常见 12 字节），文档字段名仍用 `iv` 表示「初始向量或 Nonce」的 Base64。 |
| `key` | SM4 密钥，16 字节，Base64 编码。 |
| `iv` | 与 `mode` 配套的 IV 或 GCM Nonce，Base64 编码。 |
| `ttlSeconds` | 服务端保留该 `cryptoId` 与密钥绑定的有效秒数；前端宜在超时前重新请求本接口。 |

**实现要点**

- 仅通过 **HTTPS** 返回；禁止在日志、错误信息中输出 `key` 原文或 Base64。
- `cryptoId` 对应材料在服务端 **单次消费**：校验登录请求后无论成功与否，均应使该 `cryptoId` 失效（删除 Redis 或作废记录），防止重放同一密文。
- 同一浏览器多次打开登录页可多次调用本接口；每次返回新的 `cryptoId` 与密钥材料。

**Response 错误码（可选）**

| HTTP 状态码 | 业务 `code`（字符串） | 说明 |
| --- | --- | --- |
| 200 | `"200"` | 成功 |
| 200 | `"A0215"` | 服务端暂无法签发加密参数（如熵池/依赖不可用），前端应提示稍后重试 |

### 2.2 账号密码登录

```
POST /auth/crypto/login
Content-Type: application/json
```

**Request Body**

```json
{
  "username": "string",
  "cryptoId": "string（须与 GET /auth/crypto/login 返回的一致）",
  "password": "string（SM4-CBC/PKCS7 密文的十六进制表示，偶数位；与当前后端 decryptFromHex 一致）"
}
```

**`password` 与 §2.1 `key`/`iv` 的编码约定**

| 字段 | 编码 | 说明 |
| --- | --- | --- |
| `key`、`iv`（GET 响应） | Base64 | 16 字节密钥与 IV 的 Base64 文本；前端按库约定转为二进制后参与 SM4。 |
| `password`（POST 请求体） | **Hex（十六进制）** | 明文密码经 SM4-CBC 加密后的**密文字节**再转为十六进制字符串（如 `sm-crypto` 默认输出）；**不得**再 Base64 整条密文后与后端约定混用。 |

**Response 200（登录成功）**

```json
{
  "code": "200",
  "message": "ok",
  "data": {
    "accessToken": "string",
    "expiresIn": 1800,
    "tokenType": "Bearer",
    "forceChangePassword": false
  },
  "error": null
}
```

**Set-Cookie**

```http
Set-Cookie: refreshToken={refreshToken}; HttpOnly; Secure; SameSite=Strict; Path=/api/auth/refresh
```

**Response 错误码**

| HTTP 状态码 | 业务 `code`（字符串） | 说明 |
| --- | --- | --- |
| 200 | `"200"` | 登录成功 |
| 200 | `"A0210"` | 用户名或密码错误 |
| 200 | `"A0212"` | 账号已停用，请联系管理员 |
| 200 | `"A0213"` | 登录加密参数无效、已过期或已使用（`cryptoId` 无法解析或已消费） |
| 200 | `"A0220"` | 需要强制修改密码（`data` 中仍可能含 `accessToken`，供前端进入改密流程） |

## 3. 处理流程

```
客户端                                  服务端
  │                                       │
  │  GET /auth/crypto/login               │
  │ ────────────────────────────────────> │
  │ <──────── cryptoId + key + iv + mode  │
  │                                       │
  │  POST /auth/crypto/login              │
  │ ────────────────────────────────────> │
  │                                       ├─ 1. 参数非空校验（含 cryptoId）
  │                                       ├─ 2. 解析并校验 cryptoId → 无效/过期返回 A0213，通过后取出密钥材料并置为已消费
  │                                       ├─ 3. 查询用户并校验状态 → 停用返回 A0212
  │                                       ├─ 4. 使用 SM4 将 Hex 密文 password 解密为明文
  │                                       ├─ 5. 校验密码哈希 → 失败返回 A0210
  │                                       ├─ 6. 生成 sessionId，创建会话记录
  │                                       ├─ 7. 检查密码是否需要强制修改 → 是则标记 forceChangePassword
  │                                       ├─ 8. 签发 Access Token，并通过 Set-Cookie 写入 Refresh Token
  │                                       ├─ 9. 记录登录日志（异步）
  │ <──────── Token + 用户信息             │
  │                                       │
```

## 4. 编码规范

### 4.1 密码传输

- 前端使用 **SM4**（国密对称分组密码）加密密码后传输，后端使用同一套约定解密得到明文再参与 BCrypt 校验
- 加密参数必须通过 **§2.1 `GET /auth/crypto/login`** 获取；登录请求携带 **`cryptoId`**，与返回的 `key`、`iv`、`mode` 配套使用，详见该节字段说明
- SM4 的 **密钥、IV（或 GCM Nonce）、分组模式** 由后端生成；密钥材料与 `cryptoId` **绑定且单次消费**，禁止在客户端写死长期密钥
- **密文编码（全栈须一致）**：`POST` 请求体中的 **`password` 为 SM4-CBC（PKCS#7）密文的十六进制（Hex）字符串**；`GET` 下发的 **`key`、`iv` 为 Base64**（与 §2.1 一致）。二者编码方式不同，禁止将整条密文改为 Base64 传输除非同步修改后端解析实现。
- 禁止明文传输密码

### 4.2 密码校验

- 密码比对使用 BCrypt 的 `matches` 方法，利用其内置的慢哈希和盐值机制
- 禁止自行拼接盐值后用 MD5/SHA 比对

### 4.3 Token 签发

- 使用 JWT（JSON Web Token）格式
- 签名算法：RS256（非对称）
- Access Token Payload 包含：`uid`、`username`、`sessionId`、`jti`、`tokenType: "access"`、`iat`、`exp`
- Refresh Token Payload 包含：`uid`、`sessionId`、`jti`、`tokenType: "refresh"`、`iat`、`exp`
- 禁止在 Token 中存放敏感信息（密码、权限明细等）
- 登录接口响应体中只返回 `Access Token`，`Refresh Token` 不暴露给前端 JavaScript

### 4.4 错误信息

- 登录失败时统一提示"用户名或密码错误"，不区分"用户名不存在"和"密码错误"

### 4.5 日志

- 登录成功和失败均需异步记录登录日志
- 日志字段：username、loginTime、loginIp、userAgent、loginResult（成功/失败）、failReason
- 日志写入不应阻塞登录主流程

### 4.6 前端规范

- 登录表单使用响应式校验，提交前完成客户端校验
- 登录按钮提交后需 loading 防重复点击
- 进入登录页或每次需要加密提交前，先请求 **§2.1**；若距拉取参数已超过 `ttlSeconds` 或登录返回 **`"A0213"`**（字符串），应重新请求 **§2.1** 后再提交
- 解析登录相关接口响应时，**业务成功**以 **`code === '200'`** 为准（`code` 为字符串，见 §2 总述）
- 登录成功后从 URL query 中读取 `redirect` 参数跳转，无则跳转 `/dashboard` 门户首页

## 5. 安全要点

| 项目                 | 要求                                              |
| -------------------- | ------------------------------------------------- |
| 密码传输             | SM4 加密 + HTTPS（密钥下发与单次使用策略见 §4.1） |
| 密码存储             | BCrypt（cost ≥ 10）                               |
| Refresh Token 暴露面 | 仅通过 HttpOnly Cookie 传递，前端 JS 不可读取     |
| 日志脱敏             | 日志中禁止记录密码原文或密文                      |
