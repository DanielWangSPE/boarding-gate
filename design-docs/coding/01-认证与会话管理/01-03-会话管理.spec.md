# 01-03 会话管理 — 编码规范（Spec）

## 1. 数据模型

### 1.1 会话记录（Redis Hash）

```
Key:   auth:session:{userId}
Type:  Hash
Field: {sessionId} → JSON
TTL:   与 Refresh Token 有效期一致
```

每个 Field 的 Value 结构：

```json
{
  "sessionId": "uuid",
  "userId": "string",
  "username": "string",
  "loginTime": "2026-04-01T10:00:00Z",
  "lastActiveTime": "2026-04-01T10:25:00Z",
  "loginIp": "10.0.1.100",
  "userAgent": "Mozilla/5.0 ...",
  "deviceInfo": "Windows / Chrome 120",
  "accessTokenJti": "access-token-jti",
  "refreshTokenJti": "refresh-token-jti",
  "refreshExpiresAt": "2026-04-08T10:00:00Z"
}
```

### 1.2 Token 黑名单（Redis）

```
Key:   auth:token:blacklist:{jti}
Type:  String（"revoked"）
TTL:   Token 的剩余有效时长（精确到秒）
```

## 2. 接口规范

### 2.1 刷新 Token

```
POST /auth/refresh
Cookie: refreshToken={refreshToken}（HttpOnly）
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "accessToken": "string",
    "expiresIn": 1800,
    "tokenType": "Bearer"
  }
}
```

**Response 错误码**

| 业务 code | 说明                                   |
| --------- | -------------------------------------- |
| A0230     | Refresh Token 无效或已过期，需重新登录 |
| A0231     | Refresh Token 已被吊销                 |

**实现要点**：

- Refresh Token 从 HttpOnly Cookie 中读取，不从请求体传入
- Refresh Token Payload 中必须包含 `sessionId` 和 `jti`
- 刷新时先校验会话记录是否存在，且 `refreshToken.jti` 与会话记录一致
- 刷新成功后签发新的 Access Token，Refresh Token 保持不变（不做 Rotation）
- 刷新时更新会话的 `lastActiveTime` 和 `accessTokenJti`
- 旧 Access Token 的 jti 加入黑名单（防止旧 Token 被继续使用）

### 2.2 会话心跳

```
POST /auth/session/heartbeat
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "message": "ok"
}
```

**实现要点**：

- 用于前端低频上报用户活跃状态，不要求每次交互都调用
- 距上次活跃时间更新不足 5 分钟时可直接忽略，避免高频写 Redis
- 心跳成功后仅更新当前会话的 `lastActiveTime`

## 3. 前端会话超时处理

```
用户操作（鼠标移动、键盘输入、点击、滚动）
  │
  └─ 重置空闲计时器（idleTimer）
       │
       └─ 到达 idleTimeout
            ├─ 调用主应用登出流程
            └─ 跳转登录页
```

## 4. 编码规范

### 5.1 Token 校验中间件

后端网关/过滤器的 Token 校验流程：

1. 从 `Authorization` Header 提取 Token
2. 验证 JWT 签名和 `exp` 是否过期
3. 从 Token 中提取 `jti`，检查 Redis 黑名单 `auth:token:blacklist:{jti}`
4. 黑名单命中 → 返回 401
5. 从 Token 中提取 `userId`、`sessionId`，校验会话记录是否存在
6. 会话不存在 → 返回 401
7. 会话存在 → 放行，将 userId / username / sessionId 注入请求上下文

### 5.2 活跃时间更新

- 不需要每次 API 请求都更新 `lastActiveTime`（性能开销大）
- 采用间隔更新策略：距上次更新超过 5 分钟时才更新
- 前端维护 `lastReportTime`，间隔到达后调用心跳接口或随业务请求携带 Header

## 5. 安全要点

| 项目          | 要求                                                 |
| ------------- | ---------------------------------------------------- |
| Refresh Token | HttpOnly Cookie 传递，JS 不可访问                    |
| Token 吊销    | Redis 黑名单，TTL = 剩余有效期                       |
| 会话记录      | 会话状态用于续期、超时判断和统一登出，不限制并发登录 |
