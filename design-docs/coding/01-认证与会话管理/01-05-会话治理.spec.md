# 01-05 会话治理 — 编码规范（Spec）

## 1. 数据模型

### 1.1 会话记录

会话记录复用 `01-03 会话管理` 中的 Redis 结构 `auth:session:{userId}`，每个 `sessionId` 对应一个会话实例。

## 2. 接口规范

### 2.1 指定用户全部会话失效（内部接口）

```
POST /internal/auth/users/{userId}/invalidate-sessions
```

**Request Body**

```json
{
  "reason": "USER_DISABLED"
}
```

**Response 200**

```json
{
  "code": 200,
  "message": "用户会话已全部失效"
}
```

**实现要点**：

- 仅供内部受信模块调用，如用户管理、管理员重置密码流程
- 遍历该用户所有会话，将各会话 Access Token 加入黑名单后清除会话记录
- 会话记录删除后，Refresh Token 即使仍在浏览器 Cookie 中，也无法再通过刷新接口续期

## 3. 处理流程

### 3.1 用户被停用/注销后的统一下线

```
用户管理完成停用或注销
  │
  ├─ 1. 发布领域事件 / 调用内部接口
  ├─ 2. 认证模块查询用户全部会话
  ├─ 3. 遍历加入 Access Token 黑名单
  ├─ 4. 删除全部会话记录
  └─ 5. 已登录前端在下次请求/刷新时发现会话失效并跳转登录页
```

## 4. 编码规范

### 4.1 会话失效

- 会话失效统一通过认证模块执行，其他模块不得直接删 Redis Key
- 对用户全部会话失效时，必须同时做两件事：
  1. 将当前 `accessTokenJti` 加入黑名单
  2. 删除会话记录，使 Refresh Token 无法继续刷新

### 4.2 前端处理

- 前端无需提供登录历史、在线设备或下线其他会话页面
- 已登录前端在下次请求或刷新时发现会话失效后，应统一清理登录态并跳转登录页

## 5. 安全要点

| 项目       | 要求                                                            |
| ---------- | --------------------------------------------------------------- |
| 内部接口   | 仅允许内部服务或受信后台调用                                    |
| Token 失效 | 会话删除后必须确保 Access Token 与 Refresh Token 都不可继续使用 |
| 审计追踪   | 管理员触发的批量失效需记录原因和操作人                          |
