# 04-02 角色权限配置 — 编码规范（Spec）

## 1. 数据模型

### 1.1 角色-菜单关联表（sys_role_menu）

```sql
CREATE TABLE sys_role_menu (
  role_id   BIGINT NOT NULL COMMENT '角色 ID',
  menu_id   BIGINT NOT NULL COMMENT '菜单 ID',
  PRIMARY KEY (role_id, menu_id)
) COMMENT '角色-菜单权限关联表';
```

## 2. 接口规范

### 2.1 获取角色功能权限

```
GET /roles/{id}/permissions
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "checkedMenuIds": [100, 101, 102, 200, 201],
    "halfCheckedMenuIds": [1, 10]
  }
}
```

| 字段               | 说明                                          |
| ------------------ | --------------------------------------------- |
| checkedMenuIds     | 完全勾选的菜单 ID（叶子节点 + 全选的父节点）  |
| halfCheckedMenuIds | 半选状态的菜单 ID（部分子节点被选中的父节点） |

**说明**：前端树形组件需要区分全选和半选状态才能正确回显。数据库中存储全部关联（全选 + 半选），但返回时需分开。

### 2.2 分配角色功能权限

```
PUT /roles/{id}/permissions
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "menuIds": [1, 10, 100, 101, 102, 200, 201]
}
```

**说明**：`menuIds` 包含所有选中的菜单 ID（全选 + 半选的父节点），全量替换。

**处理流程**

```
分配权限
  │
  ├─ 1. 校验角色存在且非超级管理员
  ├─ 2. 如果是子应用角色，校验 menuIds 均属于该子应用
  ├─ 3. 删除 sys_role_menu 中该角色的所有旧记录
  ├─ 4. 批量插入新的 role-menu 关联
  ├─ 5. 查询该角色关联的所有用户 ID
  ├─ 6. 批量清除用户权限缓存
  │      └─ DEL permission:user:{userId}
  │      └─ DEL menu:user:{userId}
  ├─ 7. 发布 RolePermissionChangedEvent
  ├─ 8. 记录操作留痕（含变更前后的 menuIds diff）
  └─ 9. 返回成功
```

```

**Response 200**

      { "id": 30, "deptName": "运营部" }
    ]

```

### 2.4 配置角色数据权限

```
PUT /roles/{id}/data-scope
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "dataScope": 2,
  "customDeptIds": [10, 20, 30]
}
```

**处理流程**

```
配置数据权限
  │
  ├─ 1. 更新 sys_role.data_scope
  ├─ 2. 删除 sys_role_dept 中该角色的旧记录
  ├─ 3. 如果 dataScope = CUSTOM，批量插入 sys_role_dept
  ├─ 4. 批量清除关联用户的数据权限缓存
  ├─ 5. 记录操作留痕
  └─ 6. 返回成功
```

## 3. 权限生效机制

### 3.1 用户权限集计算

```
用户的最终权限 = 所有角色权限的并集

步骤：
  1. 查询用户关联的所有角色（状态为正常）
  2. 查询这些角色关联的所有菜单 ID（sys_role_menu）
  3. 合并去重 → 用户的菜单权限集
  4. 从菜单中提取 perms 字段 → 用户的操作权限标识集
  5. 缓存到 Redis
```

### 3.2 权限缓存

| Key | 内容 | TTL |
| --- | --- | --- |
| `permission:user:{userId}` | 用户的权限标识 Set，如 `["user:list","user:create"]` | 30 分钟 |
| `menu:user:{userId}` | 用户的菜单树 JSON | 30 分钟 |
| `datascope:user:{userId}` | 用户的数据权限配置 | 30 分钟 |

**刷新时机**：

- 角色权限变更时：清除该角色所有关联用户的缓存
- 角色停用时：清除该角色所有关联用户的缓存
- 用户角色变更时：清除该用户的缓存
- 缓存自然过期后：用户下次请求时重建

### 3.3 数据权限运行时计算

```
查询请求到达
  │
  ├─ 从缓存或数据库获取用户的角色列表
  ├─ 取所有角色的 data_scope，按最大范围合并（ALL > CUSTOM > DEPT_TREE > DEPT > SELF）
  │
  ├─ 最终 scope = ALL    → 不追加过滤
  ├─ 最终 scope = CUSTOM → 合并所有角色的 customDeptIds → 追加 dept_id IN (...)
  ├─ 最终 scope = DEPT_TREE → 用户所属部门及下级 → 追加 dept_id IN (...)
  ├─ 最终 scope = DEPT   → 用户所属部门 → 追加 dept_id = ?
  └─ 最终 scope = SELF   → 追加 created_by = userId
```

## 4. 编码规范

### 4.1 全选/半选存储策略

- 数据库中存储所有关联的菜单 ID（包含全选和半选的父节点）
- 前端回显时需通过接口分别获取 `checkedMenuIds` 和 `halfCheckedMenuIds`
- 后端计算半选逻辑：如果一个父菜单的所有子菜单都在 menuIds 中 → 全选；否则 → 半选

### 4.2 超级管理员处理

- 超级管理员角色不存储具体菜单关联
- 权限判断时直接返回所有权限（`*:*:*`）
- 菜单查询时返回全部菜单（不过滤）
- 数据权限固定为 ALL

### 4.3 权限变更 diff 记录

- 操作留痕需记录变更前后的差异
- 格式：`{ added: [menuId1, menuId2], removed: [menuId3] }`
- 变更前的 menuIds 在删除前先查询出来

### 4.4 批量缓存清除

- 角色可能关联大量用户，清除缓存需高效处理
- 使用 Redis Pipeline 批量执行 DEL 命令
- 如果关联用户数超过 1000，分批处理（每批 200）

## 5. 安全要点

| 项目       | 要求                                        |
| ---------- | ------------------------------------------- |
| 权限控制   | 权限配置需 `admin:role:permission` 权限     |
| 超管保护   | 超级管理员角色的权限不可通过此接口修改      |
| 子应用隔离 | 子应用角色仅能分配该子应用的菜单            |
| 审计追踪   | 权限变更记录完整 diff，含操作前后的菜单列表 |
| 缓存一致性 | 权限变更后必须清除缓存，杜绝权限延迟生效    |
