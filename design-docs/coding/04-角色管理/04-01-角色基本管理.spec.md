# 04-01 角色基本管理 — 编码规范（Spec）

## 1. 数据模型

### 1.1 角色表（sys_role）

```sql
CREATE TABLE sys_role (
  id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
  role_name     VARCHAR(50)  NOT NULL COMMENT '角色名称',
  role_key      VARCHAR(50)  NOT NULL COMMENT '角色标识（唯一编码）',
  role_category VARCHAR(20)  NOT NULL DEFAULT 'business' COMMENT '角色分类：system/business/app',
  app_id        BIGINT       NULL     COMMENT '所属应用 ID，NULL 表示全局角色',
  data_scope    TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '数据权限范围：1-全部 2-自定义 3-本部门 4-本部门及下级 5-仅本人',
  sort_order    INT          NOT NULL DEFAULT 0 COMMENT '排序号',
  status        TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '状态：0-停用 1-正常',
  is_builtin    TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否内置角色：0-否 1-是',
  remark        VARCHAR(500) NULL     COMMENT '描述',
  created_by    BIGINT       NULL,
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by    BIGINT       NULL,
  updated_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_role_key (role_key),
  INDEX idx_app_id (app_id),
  INDEX idx_status (status)
) COMMENT '角色表';
```

### 1.2 角色数据权限范围枚举

| 值  | 常量      | 说明                   |
| --- | --------- | ---------------------- |
| 1   | ALL       | 全部数据               |
| 2   | CUSTOM    | 自定义（指定部门列表） |
| 3   | DEPT      | 本部门数据             |
| 4   | DEPT_TREE | 本部门及下级部门数据   |
| 5   | SELF      | 仅本人创建的数据       |

### 1.3 角色自定义数据权限部门表（sys_role_dept）

```sql
CREATE TABLE sys_role_dept (
  role_id   BIGINT NOT NULL COMMENT '角色 ID',
  dept_id   BIGINT NOT NULL COMMENT '部门 ID',
  PRIMARY KEY (role_id, dept_id)
) COMMENT '角色-自定义数据权限部门关联表';
```

> 仅当 `data_scope = 2（CUSTOM）` 时使用。

### 1.4 内置角色初始数据

```sql
INSERT INTO sys_role (id, role_name, role_key, role_category, data_scope, is_builtin) VALUES
(1, '超级管理员', 'super_admin', 'system', 1, 1),
(2, '普通用户',   'default',     'system', 5, 1);
```

## 2. 接口规范

### 2.1 分页查询角色列表

```
GET /roles?keyword=&category=&appId=&status=&page=1&pageSize=20
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "total": 15,
    "list": [
      {
        "id": 1,
        "roleName": "超级管理员",
        "roleKey": "super_admin",
        "roleCategory": "system",
        "appName": "全局",
        "dataScope": 1,
        "dataScopeLabel": "全部数据",
        "status": 1,
        "isBuiltin": true,
        "userCount": 2,
        "createdAt": "2026-01-01T00:00:00Z"
      }
    ]
  }
}
```

**说明**：列表中返回 `userCount`（关联用户数），通过子查询 `SELECT COUNT(*) FROM sys_user_role WHERE role_id = ?` 获取。

### 2.2 创建角色

```
POST /roles
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "roleName": "项目经理",
  "roleKey": "pm",
  "roleCategory": "business",
  "appId": null,
  "dataScope": 4,
  "customDeptIds": [],
  "sortOrder": 10,
  "remark": "项目管理角色"
}
```

**Response 错误码**

| 业务 code | 说明                                             |
| --------- | ------------------------------------------------ |
| A0500     | 角色标识已存在                                   |
| A0501     | 同一应用下角色名称已存在                         |
| A0502     | 角色标识格式错误（仅允许小写字母、数字、下划线） |

**处理流程**

```
创建角色
  │
  ├─ 1. 校验 roleKey 格式：^[a-z][a-z0-9_]{1,49}$
  ├─ 2. 校验 roleKey 全局唯一
  ├─ 3. 校验 roleName 同应用下唯一
  ├─ 4. 写入 sys_role
  ├─ 5. 如果 dataScope = CUSTOM，写入 sys_role_dept
  ├─ 6. 记录操作留痕（异步）
  └─ 7. 返回角色 ID
```

### 2.3 更新角色

```
PUT /roles/{id}
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "roleName": "高级项目经理",
  "dataScope": 2,
  "customDeptIds": [10, 20, 30],
  "sortOrder": 5,
  "remark": "更新描述"
}
```

**说明**：

- `roleKey` 不可修改，接口不接受
- 内置角色仅允许修改 `remark` 和 `sortOrder`
- 修改 `dataScope` 或 `customDeptIds` 后需清除关联用户的数据权限缓存

### 2.4 删除角色

```
DELETE /roles/{id}
Authorization: Bearer {accessToken}
```

**Response 错误码**

| 业务 code | 说明                             |
| --------- | -------------------------------- |
| A0510     | 内置角色不可删除                 |
| A0511     | 角色下存在关联用户，请先解除关联 |

**处理流程**

```
删除角色
  │
  ├─ 1. 校验非内置角色
  ├─ 2. 查询 sys_user_role 是否有关联用户 → 有则拒绝
  ├─ 3. 删除 sys_role_menu（角色-菜单关联）
  ├─ 4. 删除 sys_role_dept（角色-部门关联）
  ├─ 5. 删除 sys_role
  ├─ 6. 记录操作留痕（异步）
  └─ 7. 返回成功
```

### 2.5 启用/停用角色

```
PUT /roles/{id}/status
Content-Type: application/json
```

**Request Body**

```json
{ "status": 0 }
```

**停用处理**

```
停用角色
  │
  ├─ 1. 校验非内置角色（超级管理员不可停用）
  ├─ 2. 更新 sys_role.status = 0
  ├─ 3. 查询该角色关联的所有用户 ID
  ├─ 4. 批量清除这些用户的权限缓存
  │      └─ DEL permission:user:{userId}  （每个关联用户）
  │      └─ DEL menu:user:{userId}
  ├─ 5. 记录操作留痕（异步）
  └─ 6. 返回成功
```

## 3. 编码规范

### 3.1 角色标识规范

- 格式：`^[a-z][a-z0-9_]{1,49}$`（小写字母开头，仅含小写字母/数字/下划线）
- 创建后不可修改（更新接口忽略此字段）
- 代码中通过 `roleKey` 而非 `roleId` 判断角色，便于跨环境迁移

### 3.2 内置角色保护

- `is_builtin = 1` 的角色受保护
- 不可删除、不可停用、不可修改 `roleKey`/`roleName`/`roleCategory`
- 仅允许修改 `remark` 和 `sortOrder`
- 代码中使用常量定义内置角色 Key：

```java
public final class BuiltinRoles {
    public static final String SUPER_ADMIN = "super_admin";
    public static final String DEFAULT = "default";
}
```

### 3.3 停用联动

- 停用角色后需清除所有关联用户的权限缓存
- 通过发布 `RoleStatusChangedEvent` 事件，由权限模块监听并批量清除缓存
- 不主动踢下线（与停用用户不同），用户下次请求时重新加载权限即可

### 3.4 子应用角色

- `app_id` 不为空时为子应用角色，仅在该子应用范围内生效
- 子应用角色的菜单权限仅可分配该子应用下的菜单
- 子应用注册时可通过接口批量创建默认角色

## 4. 安全要点

| 项目       | 要求                                           |
| ---------- | ---------------------------------------------- |
| 权限控制   | 角色增删改需 `admin:role:manage` 权限          |
| 内置保护   | 内置角色硬编码保护，防止误操作                 |
| 标识不可变 | roleKey 创建后不可修改，确保代码中的引用不失效 |
| 缓存一致性 | 角色变更后批量清除关联用户的权限缓存           |
| 操作留痕   | 所有变更操作记录操作留痕                       |
