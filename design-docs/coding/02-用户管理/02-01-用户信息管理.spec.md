# 02-01 用户信息管理 — 编码规范（Spec）

## 1. 数据模型

### 1.1 用户表（sys_user）

```sql
CREATE TABLE sys_user (
  id              BIGINT       PRIMARY KEY AUTO_INCREMENT,
  username        VARCHAR(30)  NOT NULL COMMENT '用户名（登录账号）',
  password_hash   VARCHAR(128) NOT NULL COMMENT '密码哈希（BCrypt）',
  nickname        VARCHAR(50)  NOT NULL COMMENT '姓名',
  phone           VARCHAR(20)  NULL     COMMENT '手机号',
  email           VARCHAR(100) NULL     COMMENT '邮箱',
  avatar          VARCHAR(255) NULL     COMMENT '头像文件路径',
  gender          TINYINT(1)   NULL     COMMENT '性别：0-未知 1-男 2-女',
  employee_no     VARCHAR(30)  NULL     COMMENT '工号',
  user_type       VARCHAR(20)  NOT NULL DEFAULT 'internal' COMMENT '用户类型：internal/external/system',
  status          TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '状态：0-停用 1-正常 2-锁定',
  force_change_pwd TINYINT(1)  NOT NULL DEFAULT 1 COMMENT '是否需要强制修改密码',
  password_changed_at DATETIME NULL     COMMENT '最近修改密码时间',
  deleted         TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否注销：0-正常 1-已注销',
  deleted_at      DATETIME     NULL     COMMENT '注销时间',
  remark          VARCHAR(500) NULL     COMMENT '备注',
  created_by      BIGINT       NULL     COMMENT '创建人',
  created_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_by      BIGINT       NULL     COMMENT '更新人',
  updated_at      DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_username (username),
  UNIQUE INDEX uk_phone (phone, user_type),
  UNIQUE INDEX uk_email (email, user_type)
) COMMENT '系统用户表';
```

> 唯一索引中 phone 和 email 允许 NULL，MySQL 中 NULL 不参与唯一约束判断，满足"非必填但填写时唯一"的需求。

### 1.2 用户扩展字段定义表（sys_user_field）

```sql
CREATE TABLE sys_user_field (
  id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
  field_key     VARCHAR(50)  NOT NULL COMMENT '字段标识',
  field_name    VARCHAR(50)  NOT NULL COMMENT '字段显示名称',
  field_type    VARCHAR(20)  NOT NULL COMMENT '字段类型：text/number/date/select/multi_select',
  options       JSON         NULL     COMMENT '选项值（select/multi_select 类型适用）',
  required      TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否必填',
  show_in_list  TINYINT(1)   NOT NULL DEFAULT 0 COMMENT '是否在列表中展示',
  sort_order    INT          NOT NULL DEFAULT 0 COMMENT '排序号',
  status        TINYINT(1)   NOT NULL DEFAULT 1 COMMENT '状态：0-停用 1-正常',
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_field_key (field_key)
) COMMENT '用户扩展字段定义表';
```

### 1.3 用户扩展字段值表（sys_user_field_value）

```sql
CREATE TABLE sys_user_field_value (
  id            BIGINT       PRIMARY KEY AUTO_INCREMENT,
  user_id       BIGINT       NOT NULL COMMENT '用户 ID',
  field_id      BIGINT       NOT NULL COMMENT '扩展字段 ID',
  field_value   TEXT         NULL     COMMENT '字段值',
  created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_user_field (user_id, field_id),
  INDEX idx_field_id (field_id)
) COMMENT '用户扩展字段值表';
```

## 2. 接口规范

### 2.1 获取用户详情

```
GET /users/{id}
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "id": 1,
    "username": "zhangsan",
    "nickname": "张三",
    "phone": "13800138000",
    "email": "zhangsan@company.com",
    "avatar": "/files/avatars/1.jpg",
    "gender": 1,
    "employeeNo": "EMP001",
    "userType": "internal",
    "status": 1,
    "remark": "",
    "createdAt": "2026-01-01T10:00:00Z",
    "extFields": {
      "entry_date": "2024-03-15",
      "education": "本科"
    }
  }
}
```

### 2.2 更新用户信息

```
PUT /users/{id}
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "nickname": "张三",
  "phone": "13800138000",
  "email": "zhangsan@company.com",
  "gender": 1,
  "employeeNo": "EMP001",
  "remark": "",
  "extFields": {
    "entry_date": "2024-03-15",
    "education": "本科"
  }
}
```

**说明**：

- `username` 创建后不可修改，更新接口不接受此字段
- `extFields` 中只传需要更新的字段，不传的字段保持原值
- 扩展字段值按 `sys_user_field` 的定义做类型和必填校验

**Response 错误码**

| 业务 code | 说明                                   |
| --------- | -------------------------------------- |
| A0300     | 手机号已被其他用户使用                 |
| A0301     | 邮箱已被其他用户使用                   |
| A0302     | 扩展字段校验失败（附具体字段名和原因） |

### 2.3 上传用户头像

```
POST /users/{id}/avatar
Authorization: Bearer {accessToken}
Content-Type: multipart/form-data
```

**Form 字段**

| 字段 | 说明                           |
| ---- | ------------------------------ |
| file | 头像文件（jpg/png/gif，≤ 2MB） |

**Response 200**

```json
{
  "code": 200,
  "data": {
    "avatarUrl": "/files/avatars/1_1711929600.jpg"
  }
}
```

**实现要点**：

- 文件类型校验（MIME + 扩展名双重校验，防伪造）
- 文件大小限制 ≤ 2MB
- 图片处理：裁剪为正方形、压缩到合理尺寸（如 200x200）
- 文件名：`{userId}_{timestamp}.{ext}`，避免重名
- 上传成功后更新 `sys_user.avatar` 字段

### 2.4 获取当前用户信息

```
GET /users/current
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "id": 1,
    "username": "zhangsan",
    "nickname": "张三",
    "avatar": "/files/avatars/1.jpg",
    "userType": "internal",
    "roles": ["admin", "user"],
    "permissions": ["user:list", "user:create"],
    "department": {
      "id": 10,
      "name": "技术部"
    }
  }
}
```

**说明**：此接口聚合返回用户基础信息 + 角色 + 权限 + 所属部门，供前端登录后初始化使用。响应数据建议缓存（Redis，TTL 5 分钟），角色/权限变更时主动清除缓存。

## 3. 编码规范

### 3.1 字段校验规则

| 字段       | 校验规则                                |
| ---------- | --------------------------------------- |
| username   | `^[a-zA-Z0-9_]{4,30}$`，创建后不可修改  |
| nickname   | 非空，长度 1-50                         |
| phone      | 为空或符合手机号格式 `^1[3-9]\d{9}$`    |
| email      | 为空或符合邮箱格式（RFC 5322 简化版）   |
| gender     | 0/1/2 枚举值                            |
| employeeNo | 为空或长度 1-30                         |
| userType   | `internal` / `external` / `system` 枚举 |

### 3.2 扩展字段校验

- 后端需根据 `sys_user_field` 的定义动态校验扩展字段值
- `text`：字符串，长度 ≤ 500
- `number`：数值，可含小数
- `date`：日期格式 `yyyy-MM-dd`
- `select`：值必须在 `options` 列表中
- `multi_select`：值为数组，每项必须在 `options` 列表中

### 3.3 软删除（注销）

- 使用 `deleted` + `deleted_at` 字段实现软删除
- 所有查询默认加 `WHERE deleted = 0` 条件
- 注销后释放用户名：将 `username` 改为 `__deleted_{id}_{原username}`，保证唯一索引释放
- 恢复时还原 `username` 为原值

### 3.4 头像存储

- 头像文件统一存储到文件服务（本地 / MinIO）
- 数据库中存储相对路径（如 `/files/avatars/1.jpg`）
- 返回给前端时拼接文件服务域名前缀
- 上传新头像后删除旧头像文件（避免存储浪费）

### 3.5 数据权限

- 获取用户详情接口：管理员可查看所有用户，普通用户仅可查看自己
- 更新用户信息接口：管理员可编辑所有用户，普通用户仅可编辑自己的部分字段（昵称、头像等）
- 敏感字段（如手机号）在列表接口中按需脱敏

## 4. 安全要点

| 项目       | 要求                                          |
| ---------- | --------------------------------------------- |
| 头像上传   | MIME + 扩展名双重校验，限制大小，防止恶意文件 |
| 唯一性校验 | 数据库唯一索引兜底，代码层先查后写需注意并发  |
| 密码字段   | 任何查询接口均不返回 password_hash 字段       |
| 敏感字段   | 列表接口手机号等敏感信息按权限脱敏            |
