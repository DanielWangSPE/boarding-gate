# 02-03 用户关联关系 — 编码规范（Spec）

## 1. 数据模型

### 1.1 用户-部门关联表（sys_user_dept）

```sql
CREATE TABLE sys_user_dept (
  id              BIGINT    PRIMARY KEY AUTO_INCREMENT,
  user_id         BIGINT    NOT NULL COMMENT '用户 ID',
  dept_id         BIGINT    NOT NULL COMMENT '部门 ID',
  created_at      DATETIME  NOT NULL DEFAULT CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_user_dept_user (user_id),
  INDEX idx_dept_id (dept_id)
) COMMENT '用户-部门关联表';
```

### 1.2 用户-角色关联表（sys_user_role）

```sql
CREATE TABLE sys_user_role (
  id              BIGINT    PRIMARY KEY AUTO_INCREMENT,
  user_id         BIGINT    NOT NULL COMMENT '用户 ID',
  role_id         BIGINT    NOT NULL COMMENT '角色 ID',
  created_at      DATETIME  NOT NULL DEFAULT CURRENT_TIMESTAMP,

  UNIQUE INDEX uk_user_role (user_id, role_id),
  INDEX idx_role_id (role_id)
) COMMENT '用户-角色关联表';
```

## 2. 接口规范

### 2.1 查询用户所属部门

```
GET /users/{id}/departments
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "deptId": 10,
    "deptName": "技术部"
  }
}
```

### 2.2 设置用户所属部门

```
PUT /users/{id}/departments
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "deptId": 10
}
```

**校验规则**：

- `deptId` 必填
- `deptId` 必须存在且状态为正常

**Response 错误码**

| 业务 code | 说明               |
| --------- | ------------------ |
| A0340     | 必须指定所属部门   |
| A0341     | 部门不存在或已停用 |

**实现要点**：

- 采用单条覆盖策略：用户仅保留一条部门归属记录，不存在主部门概念
- 在同一事务中执行

### 2.3 查询用户角色

```
GET /users/{id}/roles
Authorization: Bearer {accessToken}
```

**Response 200**

```json
{
  "code": 200,
  "data": [
    {
      "roleId": 1,
      "roleName": "系统管理员",
      "roleKey": "admin",
      "appName": "全局"
    },
    {
      "roleId": 5,
      "roleName": "项目经理",
      "roleKey": "pm",
      "appName": "项目管理系统"
    }
  ]
}
```

### 2.4 分配用户角色

```
PUT /users/{id}/roles
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "roleIds": [1, 5, 8]
}
```

**Response 错误码**

| 业务 code | 说明               |
| --------- | ------------------ |
| A0350     | 角色不存在或已停用 |

**处理流程**

```
分配角色
  │
  ├─ 1. 校验所有 roleId 存在且状态正常
  ├─ 2. 删除 sys_user_role 中该用户的所有旧记录
  ├─ 3. 批量插入新的 user-role 关联
  ├─ 4. 清除该用户的权限缓存（Redis）
  │      ├─ 删除 permission:user:{userId}
  │      └─ 删除 menu:user:{userId}
  ├─ 5. 发布 UserRoleChangedEvent 事件
  ├─ 6. 记录操作留痕（异步）
  └─ 7. 返回成功
```

## 3. 编码规范

### 3.1 覆盖更新策略

- 部门归属接口采用单条覆盖模式
- 前端仅提交一个 `deptId`，后端更新或替换该用户唯一的部门归属记录
- 角色关联接口仍采用全量替换模式
- 注意：在事务中执行更新，避免中间状态

### 3.2 权限缓存刷新

- 角色变更后必须主动清除该用户的权限缓存
- 缓存 Key 规范：
  - `permission:user:{userId}` — 用户的权限标识集合
  - `menu:user:{userId}` — 用户的菜单树
- 清除后用户下次请求时自动重建缓存

### 3.3 事件驱动

角色变更发布事件，供其他模块消费：

```java
@Data
public class UserRoleChangedEvent {
    private Long userId;
    private String username;
    private List<Long> oldRoleIds;
    private List<Long> newRoleIds;
    private Long operatorId;
    private LocalDateTime operateTime;
}
```

监听方：

- 认证模块：可选择是否强制刷新 Token（当前方案为仅清缓存，不踢下线）
- 平台通用留痕能力：记录角色变更审计信息

### 3.4 超管角色保护

- 超级管理员角色（`roleKey = 'super_admin'`）只能由超级管理员分配
- 禁止用户移除自己的超级管理员角色（防止无超管可用）

### 3.5 并发安全

- 关联关系的全量替换有并发覆盖风险
- 在乐观锁不适用（中间表无版本号）时，使用分布式锁：
  - Lock Key：`user:bindrel:{userId}`
  - TTL：10 秒
  - 获取锁失败返回"操作频繁，请稍后重试"

## 4. 安全要点

| 项目       | 要求                                         |
| ---------- | -------------------------------------------- |
| 权限控制   | 部门/角色分配需 `admin:user:bindrel` 权限    |
| 超管保护   | 超级管理员角色的分配和移除有额外约束         |
| 缓存一致性 | 角色变更后必须清除权限缓存，防止权限延迟生效 |
| 审计追踪   | 记录完整的 old → new 变更明细，便于审计追溯  |
