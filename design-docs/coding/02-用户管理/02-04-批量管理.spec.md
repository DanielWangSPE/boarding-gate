# 02-04 批量管理 — 编码规范（Spec）

> 当前阶段不纳入用户批量导入、批量导出能力，本模块仅覆盖批量状态、角色、部门和密码操作。

## 1. 接口规范

### 1.1 批量启用/停用

```
PUT /users/batch/status
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "userIds": [1, 2, 3, 4, 5],
  "status": 0
}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "totalCount": 5,
    "successCount": 4,
    "skipped": [{ "userId": 1, "reason": "超级管理员不可操作" }]
  }
}
```

### 1.2 批量分配角色

```
PUT /users/batch/roles
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "userIds": [10, 11, 12],
  "roleIds": [2, 3],
  "mode": "append"
}
```

| mode      | 说明                           |
| --------- | ------------------------------ |
| `append`  | 追加角色（保留用户已有角色）   |
| `replace` | 替换为指定角色（覆盖已有角色） |

### 1.3 批量调整部门

```
PUT /users/batch/departments
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "userIds": [10, 11, 12],
  "deptId": 1001
}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "totalCount": 3,
    "successCount": 2,
    "skipped": [{ "userId": 12, "reason": "目标部门已停用" }]
  }
}
```

### 1.4 批量重置密码

```
POST /users/batch/reset-password
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "userIds": [10, 11, 12]
}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "totalCount": 3,
    "successCount": 3,
    "results": [
      { "userId": 10, "username": "user_a", "generatedPassword": "aB3#xY9z" },
      { "userId": 11, "username": "user_b", "generatedPassword": "kM7$nQ2w" },
      { "userId": 12, "username": "user_c", "generatedPassword": "pL5@hJ8v" }
    ]
  }
}
```

**说明**：生成的密码仅在响应中出现一次，管理员需自行保存并通知用户。

## 2. 编码规范

### 2.1 批量管理通用规则

- 批量操作中，每个用户的处理结果独立（某个失败不影响其他）
- 跳过超级管理员和操作者自身，在响应中标注跳过原因
- 批量操作需在同一事务中完成（单批 ≤ 100 条事务；超出分批提交）
- 批量操作完成后统一发布事件（如批量停用后统一发布 `UserBatchStatusChangedEvent`）

### 2.2 批量调整部门

- 批量调整部门时，目标部门必须存在且状态正常
- 用户调整部门后需同步更新 `sys_user_dept`
- 如用户是其他部门负责人，需先解除负责人身份后再调整部门

### 2.3 批量分配角色

- 角色需全部存在且状态正常
- `append` 模式追加角色，不删除原有角色
- `replace` 模式先清理旧角色，再写入新角色
- 操作完成后需清除目标用户的权限缓存

### 2.4 批量重置密码

- 批量重置密码时，为每个成功用户生成独立随机密码
- 重置成功后需调用认证模块的会话治理能力，使目标用户当前所有会话失效
- 生成密码仅在响应中返回一次，不落库保存明文

## 3. 安全要点

| 项目         | 要求                                       |
| ------------ | ------------------------------------------ |
| 批量状态权限 | 需 `admin:user:status` 权限                |
| 批量角色权限 | 需 `admin:user:bindrel` 或等效角色分配权限 |
| 批量部门权限 | 需 `admin:user:bindrel` 权限               |
| 批量密码     | 批量重置密码的结果仅返回一次，不持久化     |
