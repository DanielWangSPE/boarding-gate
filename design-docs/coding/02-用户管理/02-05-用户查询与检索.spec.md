# 02-05 用户查询与检索 — 编码规范（Spec）

## 1. 接口规范

### 1.1 分页查询用户列表

```
GET /users?keyword=张&deptId=10&roleId=1&status=1&userType=internal&createdAtFrom=2026-01-01&createdAtTo=2026-12-31&includeChildren=true&includeDeleted=false&page=1&pageSize=20&sortBy=createdAt&sortOrder=desc
Authorization: Bearer {accessToken}
```

**查询参数**

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| keyword | string | 否 | 模糊搜索（匹配用户名、姓名、手机号、邮箱） |
| deptId | long | 否 | 按部门筛选 |
| includeChildren | boolean | 否 | 部门筛选是否包含下级部门，默认 true |
| roleId | long | 否 | 按角色筛选 |
| status | int | 否 | 按状态筛选：0-停用 1-正常 2-锁定 |
| userType | string | 否 | 按用户类型筛选 |
| createdAtFrom | date | 否 | 创建时间起始 |
| createdAtTo | date | 否 | 创建时间截止 |
| includeDeleted | boolean | 否 | 是否包含已注销用户，默认 false |
| page | int | 否 | 页码，默认 1 |
| pageSize | int | 否 | 每页条数，默认 20，最大 100 |
| sortBy | string | 否 | 排序字段，默认 createdAt |
| sortOrder | string | 否 | 排序方向：asc / desc，默认 desc |

## 2. SQL 查询构建

### 2.1 基础查询模板

```sql
SELECT u.id, u.username, u.nickname, u.phone, u.email,
       u.gender, u.employee_no, u.user_type, u.status, u.created_at,
       d.id AS dept_id, d.dept_name
FROM sys_user u
LEFT JOIN sys_user_dept ud ON u.id = ud.user_id
LEFT JOIN sys_dept d ON ud.dept_id = d.id
WHERE u.deleted = 0
  -- 动态条件拼接
  AND (u.username LIKE ? OR u.nickname LIKE ? OR u.phone LIKE ? OR u.email LIKE ?)  -- keyword
  AND ud.dept_id IN (?)           -- deptId + 下级
  AND u.status = ?                -- status
  AND u.user_type = ?             -- userType
  AND u.created_at BETWEEN ? AND ?-- createdAt 范围
  -- 数据权限注入
  AND ud.dept_id IN (?)           -- 数据权限可见部门集
ORDER BY u.created_at DESC
LIMIT ?, ?
```

### 2.2 数据权限注入

数据权限过滤在 SQL 层面自动注入，通过拦截器/AOP 实现：

```
查询请求到达
  │
  ├─ 从 Token 中获取当前用户 userId
  ├─ 查询该用户的数据权限范围（角色上配置的 dataScope）
  │
  ├─ dataScope = ALL         → 不追加条件
  ├─ dataScope = DEPT        → 追加 AND ud.dept_id IN (用户所属部门ID)
  ├─ dataScope = DEPT_TREE   → 追加 AND ud.dept_id IN (用户部门及所有下级部门ID)
  ├─ dataScope = SELF        → 追加 AND u.id = 当前userId
  └─ dataScope = CUSTOM      → 追加 AND ud.dept_id IN (角色配置的自定义部门ID列表)
```

### 2.3 部门下级查询

筛选部门包含下级时，需获取指定部门的所有下级 ID：

```sql
-- 方案1：使用部门路径前缀匹配（推荐，需在 sys_dept 中维护 ancestors 字段）
SELECT id FROM sys_dept WHERE ancestors LIKE CONCAT(?, '%') OR id = ?

-- 方案2：递归 CTE（MySQL 8.0+）
WITH RECURSIVE dept_tree AS (
  SELECT id FROM sys_dept WHERE id = ?
  UNION ALL
  SELECT d.id FROM sys_dept d JOIN dept_tree dt ON d.parent_id = dt.id
)
SELECT id FROM dept_tree
```

推荐方案 1，`sys_dept.ancestors` 存储完整祖先路径如 `0,1,10,`，查询效率更高。

## 3. 编码规范

### 3.1 分页参数约束

- `page` 最小值 1
- `pageSize` 范围 1-100，超出范围自动修正为边界值
- `sortBy` 仅允许白名单字段：`createdAt`、`username`、`nickname`、`status`，防止 SQL 注入
- `sortOrder` 仅允许 `asc` / `desc`

### 3.2 模糊搜索

- 使用 `LIKE %keyword%` 进行模糊匹配
- keyword 需做特殊字符转义（`%`、`_`、`\`）
- keyword 长度限制 1-50 字符
- 多字段模糊搜索使用 OR 连接，单个 keyword 同时匹配多个字段

### 3.3 列表脱敏

- 手机号脱敏：`138****8000`（显示前三后四）
- 邮箱脱敏：`zha****@company.com`（@前保留前三位）
- 脱敏在 Service 层统一处理，提供 `@Desensitize` 注解标记需脱敏字段
- 用户详情接口不脱敏（管理员查看完整信息）

### 3.4 查询性能

- 确保常用查询条件有索引覆盖
- `keyword` 模糊搜索在大数据量下考虑使用全文索引或 Elasticsearch
- 部门下级 ID 列表可缓存（部门树变更时刷新）
- 用户列表接口的角色信息通过二次查询批量获取，避免 N+1 问题

### 3.5 总数查询优化

- 分页查询同时返回 `total`（总数）
- 大表场景下 `COUNT(*)` 可能较慢，可使用近似值或缓存策略
- 当 `page > 1` 且前端已有 `total` 时，可通过请求参数标记跳过 COUNT

## 4. 安全要点

| 项目     | 要求                                          |
| -------- | --------------------------------------------- |
| SQL 注入 | sortBy 字段白名单校验；所有参数使用参数化查询 |
| 数据权限 | 查询结果必须受数据权限范围约束                |
| 敏感数据 | 列表接口敏感字段脱敏；密码字段永不返回        |
| 参数校验 | pageSize 限制上限防止恶意大量查询             |
| 权限控制 | 用户列表接口需 `user:list` 权限               |
