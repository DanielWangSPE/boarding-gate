# 02-02 用户生命周期 — 编码规范（Spec）

## 1. 接口规范

### 1.1 创建用户

```
POST /users
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "username": "zhangsan",
  "nickname": "张三",
  "password": "string（RSA 加密密文，可选。为空则系统自动生成）",
  "phone": null,
  "email": null,
  "gender": 1,
  "employeeNo": "EMP001",
  "userType": "internal",
  "departmentIds": [10],
  "primaryDepartmentId": 10,
  "roleIds": [1, 2],
  "remark": ""
}
```

**Response 200**

```json
{
  "code": 200,
  "data": {
    "id": 100,
    "username": "zhangsan",
    "generatedPassword": "xK9#mP2q（仅自动生成时返回）"
  },
  "message": "创建成功"
}
```

**Response 错误码**

| 业务 code | 说明                     |
| --------- | ------------------------ |
| A0310     | 用户名已存在             |
| A0311     | 手机号已被使用           |
| A0312     | 邮箱已被使用             |
| A0313     | 密码不符合复杂度要求     |
| A0314     | 关联的部门不存在或已停用 |
| A0315     | 关联的角色不存在或已停用 |

**处理流程**

```
接收创建请求
  │
  ├─ 1. 参数校验（格式、必填项）
  ├─ 2. 唯一性校验（username / phone / email）
  ├─ 3. 密码处理
  │      ├─ 有密码 → RSA 解密 → 复杂度校验 → BCrypt 哈希
  │      └─ 无密码 → 按策略生成随机密码 → BCrypt 哈希
  ├─ 4. 写入 sys_user（status=1, force_change_pwd=1）
  ├─ 5. 写入用户-部门关联（sys_user_dept）
  ├─ 6. 写入用户-角色关联（sys_user_role）
  ├─ 7. 记录操作留痕（异步）
  └─ 8. 返回创建结果
```

### 1.2 启用/停用用户

```
PUT /users/{id}/status
Authorization: Bearer {accessToken}
Content-Type: application/json
```

**Request Body**

```json
{
  "status": 0
}
```

| status | 说明 |
| ------ | ---- |
| 0      | 停用 |
| 1      | 启用 |

**Response 错误码**

| 业务 code | 说明                   |
| --------- | ---------------------- |
| A0320     | 不可操作超级管理员账号 |
| A0321     | 不可操作自己的账号     |

**停用处理流程**

```
停用用户
  │
  ├─ 1. 权限校验（需 admin:user:status 权限）
  ├─ 2. 目标用户校验
  │      ├─ 是否超级管理员 → 是则拒绝
  │      └─ 是否自己 → 是则拒绝
  ├─ 3. 更新 sys_user.status = 0
  ├─ 4. 更新后续登录校验状态，使停用账号不可再登录
  ├─ 5. 记录操作留痕（异步）
  └─ 6. 返回成功
```

### 1.3 解锁用户

```
POST /users/{id}/unlock
Authorization: Bearer {accessToken}
```

**实现要点**：

- 清除 Redis 中的登录失败计数 `auth:login:fail:{username}`
- 清除 Redis 中的锁定标记 `auth:login:lock:{username}`
- 如果 `sys_user.status = 2（锁定）`，更新为 `1（正常）`
- 需要 `admin:user:unlock` 权限

### 1.4 注销用户

```
DELETE /users/{id}
Authorization: Bearer {accessToken}
```

**Response 错误码**

| 业务 code | 说明                   |
| --------- | ---------------------- |
| A0320     | 不可操作超级管理员账号 |
| A0321     | 不可操作自己的账号     |

**处理流程**

```
注销用户
  │
  ├─ 1. 权限及目标用户校验（同停用）
  ├─ 2. 更新 sys_user：
  │      ├─ deleted = 1
  │      ├─ deleted_at = NOW()
  │      └─ username → __deleted_{id}_{原username}（释放唯一索引）
  ├─ 3. 调用会话管理：踢下线 + Token 黑名单
  ├─ 4. 记录操作留痕（异步）
  └─ 5. 返回成功
```

### 1.5 恢复已注销用户

```
POST /users/{id}/restore
Authorization: Bearer {accessToken}
```

**Response 错误码**

| 业务 code | 说明                   |
| --------- | ---------------------- |
| A0330     | 用户不在已注销状态     |
| A0331     | 已超过可恢复期限       |
| A0332     | 原用户名已被新用户占用 |

**处理流程**

```
恢复用户
  │
  ├─ 1. 检查用户是否处于已注销状态
  ├─ 2. 检查是否在可恢复期限内（deleted_at + recoverableDays）
  ├─ 3. 提取原用户名（从 __deleted_{id}_{原username} 中解析）
  ├─ 4. 检查原用户名是否已被其他用户占用
  ├─ 5. 还原 username、deleted=0、deleted_at=NULL
  ├─ 6. 设置 status=1, force_change_pwd=1（恢复后强制改密）
  ├─ 7. 记录操作留痕（异步）
  └─ 8. 返回成功
```

## 2. 配置项

| 配置项                         | 说明                         | 默认值 |
| ------------------------------ | ---------------------------- | ------ |
| `user.deleted.recoverableDays` | 注销后可恢复天数，0=不可恢复 | 30 天  |

## 3. 编码规范

### 3.1 状态枚举

```java
public enum UserStatus {
    DISABLED(0, "停用"),
    NORMAL(1, "正常"),
    LOCKED(2, "锁定");
}
```

### 3.2 事务边界

- 创建用户：用户表 + 部门关联 + 角色关联在同一事务中
- 停用/注销用户：数据库更新和 Redis 操作分离——数据库在事务中完成，Redis（踢下线）在事务提交后异步执行
- 如果 Redis 操作失败，不回滚数据库事务，而是记录异常日志并重试

### 3.3 停用联动

- 停用用户时调用认证模块的会话失效接口，保持模块间松耦合
- 推荐通过 Spring Event 发布 `UserStatusChangedEvent`，由认证模块监听并处理
- 事件数据：`{ userId, username, oldStatus, newStatus, operator, operateTime }`

### 3.4 初始密码

- 系统自动生成密码的逻辑复用密码管理模块的 `PasswordGenerator`
- `generatedPassword` 仅在创建响应中返回一次，后端不做持久化存储
- 管理员需自行将初始密码告知用户

### 3.5 幂等性

- 重复停用已停用的用户：直接返回成功，不报错
- 重复启用已启用的用户：直接返回成功，不报错
- 注销已注销的用户：返回"用户不存在"（因为常规查询默认排除已注销用户）

## 4. 安全要点

| 项目     | 要求                                                              |
| -------- | ----------------------------------------------------------------- |
| 超管保护 | 任何状态变更操作均需校验目标用户是否为超级管理员                  |
| 自我保护 | 禁止对自身执行停用、注销操作                                      |
| 密码安全 | 自动生成的密码仅返回一次，不持久化；手动指定的密码需 RSA 加密传输 |
| 操作留痕 | 创建、停用、启用、解锁、注销、恢复操作均需记录操作留痕            |
| 会话清理 | 停用/注销后确保所有 Token 失效，防止状态变更后仍可访问            |
