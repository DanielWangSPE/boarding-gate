# 07 - 微前端接入机制

## 1. 模块概述

微前端接入机制是统一门户平台的架构核心，负责定义子应用的接入标准、运行时管理和应用间协作机制。通过微前端架构，各业务系统可以独立开发、独立部署，同时在门户平台的框架内以统一体验呈现给用户。

## 2. 架构设计

### 2.1 整体架构

```
┌───────────────────────────────────────────────────────┐
│                   统一门户（主应用）                      │
│  ┌─────────────────────────────────────────────────┐  │
│  │  顶部导航栏 / 侧边菜单 / 面包屑 / 标签页         │  │
│  ├─────────────────────────────────────────────────┤  │
│  │                                                 │  │
│  │           子应用渲染容器区域                       │  │
│  │                                                 │  │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐      │  │
│  │  │ 子应用 A  │  │ 子应用 B  │  │ 子应用 C  │      │  │
│  │  │ (Vue)    │  │ (React)  │  │ (Angular) │      │  │
│  │  └──────────┘  └──────────┘  └──────────┘      │  │
│  └─────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────┘
```

### 2.2 技术选型：micro-app

经综合对比，选定 **micro-app**（京东开源）作为微前端框架。

| 维度 | qiankun | **micro-app** ✅ | wujie | Module Federation |
| --- | --- | --- | --- | --- |
| 底层原理 | single-spa + Proxy 沙箱 | **Web Components + Proxy 沙箱** | iframe 沙箱 + Web Components | Webpack/Vite 模块共享 |
| 接入成本 | 中等 | **极低** | 低 | 高 |
| CSS 隔离 | scopedCSS（有边界 case） | **Shadow DOM** | iframe 天然隔离 | 无 |
| 内置 keep-alive | 无 | **有** | 有 | 无 |
| 子应用改造量 | 中等 | **最少** | 少 | 多 |

**选型理由**：接入成本最低（自定义标签）、隔离均衡（Proxy + Shadow DOM，无 iframe 弹窗问题）、内置 keep-alive（适配多标签页）、通信机制完善、与 Vue 主应用天然契合。

> 详细对比与代码示例参见各子模块 spec 文档。

### 2.3 基于 micro-app 的接入实现方式

基于 `micro-app` 的微前端接入，核心上由主应用负责注册、挂载、路由分发和公共能力下发，子应用按统一规范完成入口适配和生命周期管理。

#### 2.3.1 子应用注册

- 平台维护子应用注册信息，包括 `appId`、`appName`、`entryUrl`、`routePrefix` 等
- `appId` 全局唯一，作为子应用在门户中的唯一标识
- `routePrefix` 全局唯一，且不得与门户内置路由冲突
- 子应用注册完成后，平台才能进行菜单关联、路由激活和运行状态管理

#### 2.3.2 主应用挂载

- 主应用在入口处初始化 `micro-app`
- 页面内容区通过 `<micro-app>` 自定义元素挂载子应用
- 挂载时传入子应用名称、入口地址、基础路由、缓存策略等信息
- 主应用统一处理子应用的加载中、加载失败、维护中等状态

#### 2.3.3 路由分发

- 主应用根据当前访问路径识别是否命中某个子应用的 `routePrefix`
- 命中后，主应用激活对应子应用容器并渲染子应用
- 子应用内部路由需基于主应用下发的基础路由运行
- 子应用既要支持接入门户运行，也要支持脱离门户独立运行

#### 2.3.4 公共能力下发

- 主应用通过 `micro-app` 数据通信机制向子应用下发公共运行数据
- 下发内容包括用户信息、登录态、权限数据、主题配置等
- 子应用不得自行维护独立的一套认证和权限体系，应优先消费主应用下发的数据
- 公共能力变更时，主应用可继续向已加载子应用增量同步

#### 2.3.5 子应用事件回传

- 子应用通过事件通信向主应用发送一次性事件
- 典型事件包括路由变化、页面标题变化、异常上报、请求登出等
- 主应用负责接收事件并执行统一处理，如更新浏览器地址、更新标签页标题、执行全局登出流程
- 子应用间通信原则上通过主应用中转，不直接互调

#### 2.3.6 生命周期与清理

- 子应用需适配创建、挂载、卸载等生命周期
- 卸载时必须清理定时器、全局事件、WebSocket 连接等副作用
- 主应用负责感知子应用加载失败和运行时异常，并提供统一兜底
- 对常用子应用可启用 keep-alive，以提升切换体验

#### 2.3.7 隔离机制

- JS 运行环境通过 Proxy 沙箱隔离，避免子应用污染全局对象
- CSS 默认通过 Shadow DOM 隔离，避免样式串扰
- 浏览器存储按应用维度隔离，避免不同子应用之间相互覆盖
- 如确有必要暴露全局变量或共享依赖，必须通过白名单或统一 SDK 方式接入

## 4. 业务规则汇总

| 编号 | 规则 | 说明 | 所属模块 |
| --- | --- | --- | --- |
| BR-01 | 应用标识唯一 | 子应用标识全局唯一，注册后不可修改 | 子应用注册与管理 |
| BR-02 | 路由前缀唯一 | 各子应用的激活路由前缀不可重复 | 子应用注册与管理 |
| BR-03 | 故障隔离 | 子应用异常不得影响主应用和其他子应用的正常运行 | 子应用生命周期与沙箱 |
| BR-04 | 样式隔离 | 子应用样式变化不得影响主应用和其他子应用的样式 | 子应用生命周期与沙箱 |
| BR-05 | 独立运行 | 子应用应能脱离主应用独立运行（用于开发和调试） | 公共依赖与接入规范 |
| BR-06 | 版本兼容 | 子应用版本更新不应导致主应用框架需要同步更新 | 公共依赖与接入规范 |
| BR-07 | 健康检查 | 子应用入口地址不可用时自动标记为异常状态 | 子应用注册与管理 |
