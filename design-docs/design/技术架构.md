# 统一门户平台（Boarding Gate）— 技术架构

## 1. 文档目的与适用范围

### 1.1 目的

本文档定义 **统一门户平台** 在实现层面的技术选型、分层结构、接口约定与 UI 约束，作为后续功能开发、代码评审与部署运维的 **强制性技术基线**。当业务需求文档与本文档冲突时，应先修订本文档或需求文档并达成一致后再开发。

### 1.2 适用范围

- 主应用（门户壳层、认证、菜单、门户首页、系统管理后台等）
- 与主应用对接的 **Java 8 + Spring Boot 后端服务** 及 **MySQL 8 持久化**
- 子应用接入方式在技术选型上需与主应用协调（见第 7 节）

## 2. 架构总览

### 2.1 前后端分离

采用 **前后端分离**：浏览器仅与静态前端资源及后端 HTTP(S) API 交互；**禁止** 在浏览器直连数据库或内网仅暴露给后端的端口。

```
┌─────────────────────────────────────────────────────────────┐
│                        浏览器                                 │
│  Vue 3 主应用（布局 / 路由 / 权限 / 微前端容器）               │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS / JSON API + WebSocket(可选)
┌───────────────────────────▼─────────────────────────────────┐
│        Java 8 + Spring Boot 应用服务（单实例或多实例）         │
│  REST/JSON · 认证鉴权 · 业务编排 · 文件与任务等               │
└───────────────────────────┬─────────────────────────────────┘
                            │ JDBC / 文件
┌───────────────────────────▼─────────────────────────────────┐
│  MySQL 8 · 连接池 · 迁移脚本（Flyway/Liquibase 等）· 备份策略 │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 与需求定位的对应关系

| 需求总览中的定位 | 架构落点 |
| --- | --- |
| 企业级统一门户 + 微前端主应用 | 前端采用可承载子应用的壳工程；路由与菜单由后端驱动时与权限模型一致 |
| 私有化部署 | 前后端均可容器化或传统进程部署；MySQL 连接与库名等通过配置注入 |
| 统一认证 / 授权 / 菜单 | 后端签发令牌与权限声明；前端路由守卫与按钮级指令与之一致 |

## 3. 前端技术架构

### 3.1 技术栈（强制）

| 层级 | 选型 | 说明 |
| --- | --- | --- |
| 框架 | **Vue 3**（Composition API 为主） | `<script setup>` 与组合式 API 为默认写法 |
| 构建 | **Vite** | 开发与生产构建工具 |
| 语言 | **TypeScript** | 业务代码强类型；`any` 仅用于边界兼容并需注释原因 |
| 路由 | **Vue Router** | 支持动态路由与权限路由 |
| 状态 | **Pinia** | 全局状态；避免滥用全局状态，优先 props/注入 |
| HTTP | **Axios**（或经框架封装的等价客户端） | 统一拦截器：Token、错误码、刷新与登出 |
| 样式 | **CSS 变量 + 可选 UnoCSS/Tailwind**（与所选 UI 基座一致） | 主题与暗色模式与 UI 基座对齐 |

### 3.2 UI 与交互：对齐 Vben Admin

**参考站点**：[Vben Admin 演示](https://www.vben.pro/#/analytics)（以当前线上演示为准）。

**目标**：在布局结构、导航形态、表格/表单密度、主题与动效等方面 **尽可能与 Vben Admin 官方演示一致**（下称「视觉与交互基线」）。实现策略按优先级如下。

| 策略 | 说明 | 还原度预期 |
| --- | --- | --- |
| **A. 基于官方模板工程（推荐）** | 采用与 Vben 生态一致的 **vue-vben-admin**（或官方文档推荐的等价 starter），在同一套布局、主题与组件约定上扩展业务模块 | 最高，可持续跟随上游修复与升级 |
| **B. 自研壳层** | 自行实现侧边栏、顶栏、多 Tab、面包屑、内容区等；**必须**建立与 Vben 对齐的设计 Token（颜色、圆角、间距、阴影、字体）并固定组件库版本 | 依赖设计稿与逐页对照，工作量与偏差风险较高 |

**约束（强制）**：

1. **组件库**：与所选 Vben 版本一致（通常为 **Ant Design Vue**；若模板升级导致变更，以模板锁定版本为准）。
2. **布局**：经典后台布局（可折叠侧栏、顶栏用户区与设置、内容区多页签/面包屑等）与演示站 **同结构同交互习惯**，不得替换为明显不同风格的其它 Admin 模板。
3. **主题**：支持亮色/暗色（及演示站具备的其它主题能力时一并支持）；主题变量命名与覆盖方式遵循模板工程规范。
4. **页面级还原**：业务页面（如分析页、表格页、表单页）在 **栅格、卡片、图表占位、空状态** 上与演示站同类页面 **像素级尽量一致**；自定义业务内容仅替换数据与文案区域。
5. **图标与资源**：优先使用模板内置图标体系与资源路径约定，避免混用多套图标风格。

**说明**：第三方演示站可能随版本更新微调样式；**以项目锁定的模板版本 + 设计 Token 文档** 为验收依据，线上演示为交叉参照。

### 3.3 工程与目录约定（建议）

- **按领域分包**：`views`、`api`、`stores`、`router`、`components` 等与业务模块对应，便于与 `docs/` 中模块编号对齐。
- **环境变量**：`VITE_*` 前缀；区分 `development` / `production`；API 基地址不得写死在业务代码中。
- **代码规范**：ESLint + Prettier（或与模板一致）；提交前可配合 lint-staged。

### 3.4 与后端的协作方式

- 默认 **JSON** 通信；时间统一 **ISO 8601** 字符串（UTC 或带时区，由接口文档明确）。
- 分页、排序、筛选参数命名与后端 **统一约定一套**（建议在《接口规范》中成文）。
- 上传下载走后端提供的 URL 或流式接口，前端不绕过鉴权。

## 4. 后端技术架构

### 4.1 技术栈（强制）

| 项目 | 选型 | 说明 |
| --- | --- | --- |
| 运行时 | **JDK 8** | 以项目 `JAVA_HOME`、构建脚本或 CI 声明为准；**禁止** 在无一致升级决议的情况下混用更高 JDK 编译/运行 |
| Web 框架 | **Spring Boot 2.x**（与 JDK 8 兼容，具体小版本项目锁死） | 统一依赖管理与自动配置；模块划分采用典型 Spring 分层与包结构 |
| 语言 | **Java 8** | 语法与 API 以 Java 8 为上限（除非另有升级基线文档） |
| Web | **Spring MVC**（随 Spring Boot） | REST 控制器、消息转换、全局异常处理 |
| 校验 | **Bean Validation**（`javax.validation`，随 Spring Boot 2.x）+ `spring-boot-starter-validation` | 入参校验在服务端执行；DTO 上使用约束注解 |
| 持久化 | **Spring Data JPA** 或 **MyBatis（含 MyBatis-Plus 等）**（二选一，项目启动时锁定） | 必须支持迁移脚本与可重复的 schema 演进 |
| 数据库 | **MySQL 8** | 驱动采用官方 **MySQL Connector/J**；字符集建议 **utf8mb4**，排序规则与业务约定一致 |
| 连接池 | **HikariCP**（Spring Boot 默认） | 连接数、超时等通过配置调优，禁止在业务代码中裸建连接 |

### 4.2 分层结构（强制意图）

```
Controller（Web 层）
    → Service（应用服务层：用例编排、事务边界）
    → Repository / Mapper（持久化：与表结构、查询封装）
    → MySQL 8
```

- **禁止** 在 Controller 中写复杂 SQL 或跨表事务散落多处；事务边界在 Service 层通过 `@Transactional` 等方式显式表达。
- **横切**：认证、鉴权、日志、异常到 HTTP 状态码的映射，通过过滤器、拦截器、`@ControllerAdvice` 等统一处理。

### 4.3 API 形态

- **REST** 为主，资源名词与 HTTP 动词语义清晰；必要时 **RPC 风格** 的少数动作端点需单独成文说明。
- **版本**：建议 URL 前缀 `/api/v1`，破坏性变更递增主版本。
- **错误体**：统一结构，例如 `{ code, message, details?, traceId? }`，`traceId` 便于与日志关联。
- **OpenAPI**：推荐通过 **springdoc-openapi** 或 **Spring REST Docs** 等维护 OpenAPI 3 描述，便于前端生成类型与联调。

### 4.4 认证与授权（与需求模块对齐）

- **登录后令牌**：建议使用 **JWT（访问令牌 + 刷新令牌）** 或 **服务端 Session + HttpOnly Cookie**；二者选其一并在全项目贯彻。
- **权限模型**：与 RBAC/ABAC 需求一致；接口层做 **权限校验**，不信任前端隐藏按钮。
- **密码存储**：强哈希（如 bcrypt/argon2）；禁止明文或可逆存储。

### 4.5 实时能力

- 消息通知、在线会话等若需推送，可采用 **WebSocket** 或 **SSE**；连接同样需要认证与授权。

### 4.6 文件与异步任务

- 文件上传：限制类型与大小；存储路径可配置；元数据在 MySQL 记录。
- 异步任务：如确有平台内部维护任务，可使用 **Spring `@Async` / 定时任务**、消息队列（若引入）或 **独立 Worker 进程**；避免在 Web 线程中执行长耗时阻塞任务。

## 5. 数据库与数据架构（MySQL 8）

### 5.1 使用原则

- **版本与引擎**：数据库版本 **MySQL 8**；默认存储引擎 **InnoDB**；时区、字符集、排序规则在环境与配置中显式约定。
- **连接与池化**：应用通过 **HikariCP** 访问数据库；禁止在请求处理中频繁创建/销毁物理连接。
- **迁移**：所有表结构变更 **必须** 通过 **Flyway**、**Liquibase** 或项目统一认可的迁移机制版本化管理，禁止生产手工改表。
- **SQL 与索引**：避免 N+1 与全表扫描；关键查询需有索引与执行计划评审；禁止未经评审的字符串拼接 SQL（防注入）。

### 5.2 备份与恢复

- 定期逻辑备份（如 `mysqldump`）或按运维规范做物理备份；重大变更前强制备份。
- 文档化恢复步骤（含时间点恢复策略若适用），满足审计与灾备最低要求。

## 6. 部署与运行

### 6.1 典型拓扑（私有化）

- **静态资源**：Nginx（或同类）托管前端 `dist`。
- **API**：Spring Boot 可执行包（JAR）或容器内 JVM 进程监听内网端口，由反向代理对外；仅暴露 HTTPS。
- **MySQL**：独立部署或托管实例；网络与账号权限最小化，应用账号遵循最小权限原则。

### 6.2 配置

- 使用环境变量、`application.yml` / `application-*.yml`（**不入库**）管理密钥、数据源 URL、用户名等敏感配置。
- **禁止** 将密钥与生产口令提交到版本库。

## 7. 微前端（与子应用）

需求总览要求主应用承担 **微前端容器** 职责。与本技术栈的衔接约束如下：

| 项 | 约束 |
| --- | --- |
| 主应用技术 | Vue 3；与 Vben 模板使用的微前端方案 **保持一致**（常见为 **qiankun** 或与官方模板一致方案） |
| 子应用 | 可为 Vue/React 等，须满足接入规范（见 `docs/07-微前端接入机制`） |
| 样式隔离 | 遵循模板与 qiankun（或选定方案）的约定，避免全局污染 |
| 通信 | 与 `docs/07` 中事件总线 / props / 共享状态方案一致 |

若首期仅交付「壳 + 后端」而无真实子应用，仍应按上述方案预留入口与配置结构。

## 9. 待决事项（可选后续补充）

以下可在启动迭代前收口为决议并更新本文档：

1. **Spring Boot** 具体小版本与 **Spring Cloud**（若需要）是否引入及版本矩阵。

5. 前端是否 **fork vue-vben-admin 固定版本** 作为唯一基线（强烈建议为「是」以利于 UI 还原）。
6. OpenAPI 是否以 **springdoc-openapi** 注解驱动为主，或辅以手写 YAML 约束。
